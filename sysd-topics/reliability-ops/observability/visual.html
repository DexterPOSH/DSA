Observability — Logs, Metrics & Traces

How the three pillars let you infer a system's internal state from the outside — and how a single shared trace_id stitches them together so a 3am pager turns into a root cause in minutes.

Real-world analogy — monitoring a hospital patient

METRICS = the bedside monitor showing heart rate, BP and oxygen as live numbers. It instantly screams "something's wrong" (BP dropped!) but never says why. Cheap, continuous, always on.

LOGS = the nurse's diary, every event timestamped: "2:04pm gave medicine X", "2:11pm patient complained of pain". Rich detail to reconstruct exactly what happened — but storing every patient's full diary is expensive.

TRACES = the patient file that travels ER → X-ray → surgery → ICU, recording time spent at each stop. It reveals the bottleneck was the X-ray queue, not surgery — i.e. where the delay was.

No single tool tells the whole story. The metric shouts there's a problem, the trace says where, the log says exactly what — observability is fusing all three to diagnose problems you never anticipated.

Metrics

"Is something wrong, right now?"
http_requests_total{svc="pay"}
request_latency_ms p99=820
Aggregate numbers in a time-series DB (Prometheus). Cost scales with cardinality, not request volume — 1M requests = 1 counter bump. Cheapest pillar, runs at 100% traffic, powers dashboards & alerts. Never put user_id here (cardinality explosion).

Logs

"What exactly happened?"
{"ts":...,"level":"ERROR",
"user_id":42,"trace_id":"a1b2"}
Discrete timestamped events, shipped to ELK / Loki / Splunk. Use structured (JSON) logging so machines can query. Volume dominates cost at scale → tame with log levels, sampling, and retention tiers.

Traces

"Where did the time go?"
trace a1b2 → tree of spans
gateway · auth · pay · db
End-to-end request journey as a tree of spans (Jaeger / Tempo / Zipkin). A trace_id + parent span_id ride HTTP headers to each hop (context propagation). Most expensive → use sampling.

Things to know in the interview

  • Monitoring ≠ Observability. Monitoring answers pre-defined questions ("is CPU high?"). Observability lets you ask unknown ones ("which tenant with feature-flag X on is slow?") without deploying new code. High-cardinality traces + structured logs give that explorability.
  • Cardinality is the metric killer. Putting user_id, request_id or trace_id as a Prometheus label spawns one series per value → millions of series → TSDB OOM. Unique-per-request data belongs in logs/traces, not metrics.
  • Context propagation — the entry point mints a trace_id; every downstream call forwards it + the parent span_id via the W3C traceparent header. Without it, 30 services' logs can't be correlated.
  • Sampling = deliberate data loss. Head-based decides at entry (cheap, can miss rare errors). Tail-based buffers the whole trace then keeps all errors/slow ones (pricier, needs a central collector). Bias toward keeping anomalies.
  • Push vs pull — Prometheus pulls (scrapes); short-lived batch jobs need a Pushgateway. StatsD / OTel push, better for ephemeral workloads.
  • OpenTelemetry (OTel) is the vendor-neutral standard: one SDK emits traces/metrics/logs to any backend — avoids lock-in at the cost of running a collector layer.
  • Clock skew — trace waterfalls depend on different machines' wall clocks, so span alignment can be slightly off. trace_id parent/child links are causal and immune; read the timeline with a grain of salt.

Key takeaway

Emit all three pillars and correlate them with a shared trace_id: a metric alert tells you something broke, a trace waterfall tells you where the latency went, and the logs for that exact trace_id tell you why. Keep high-cardinality data out of metrics, sample traces/logs to control cost, and instrument once with OpenTelemetry.
