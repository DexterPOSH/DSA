Rate Limiting — The Token Bucket

How a service caps each client to a fixed rate, deciding in microseconds whether to allow a request or reject it with 429 Too Many Requests — while still absorbing a controlled burst.

Real-world analogy — the nightclub bouncer

A popular club has a bouncer whose rule is: "Max 60 people in per minute" — on average one per second. If 200 people rush the gate in a single second, the bouncer won't let them all in: some wait, some get turned away (rejected), even if there's room inside. That's rate limiting — we deliberately cap the flow so the backend (DB, downstream services) gets a predictable, controlled stream and no single aggressive client ruins it for everyone. The token bucket bouncer is a bit flexible: if the last few minutes were quiet, he's let "saved-up" capacity build, so he can wave a small burst through at once.

Fixed Window — boundary burst

limit = 100 / min, counter resets each window

Counter is fine inside a window, but it resets at the edge. A client fires 100 requests at 10:00:59 and 100 more at 10:01:00 — straddling the reset. Result: 200 requests pass in ~1 second — effectively 2× the limit.

Cheap (one integer per key) but gives no real per-second guarantee. The classic interview gotcha.

Sliding Window Counter — the fix

est = prev × (1 − elapsed%) + curr

Blends the previous and current window by how far into the current one you are. Limit 100/min, 40% into the window, prev = 90, curr = 20 → est = 90 × 0.6 + 20 = 74. 74 < 100 → allow. Smooths the boundary with just two counters per key.

Approximate, but production-grade (Cloudflare-style) accuracy at tiny memory cost.

Things to know in the interview

  • Token Bucket ≠ Leaky Bucket. Token bucket allows bursts (tokens accumulate up to B). Leaky bucket smooths output to a constant rate — no burst. Interviewers probe this distinction directly.
  • Distributed = shared, atomic state. With 100s of gateway nodes, counters live in a central fast store — typically Redis (sub-ms). Use atomic INCR or a Lua script so a concurrent read → check → write can't let two requests both see "99 < 100" and slip through (lost update).
  • Local vs centralized. Local in-memory counters are fast (zero hops) but each node tracks its own limit → effective global cap becomes N_nodes × limit (too permissive). Centralized Redis is accurate but a round-trip per request and a potential bottleneck. Hybrid = local cache + async sync.
  • Fail-open vs fail-closed. If the limiter/Redis dies, do you allow (fail-open, availability-first — most public APIs) or block (fail-closed, protection-first — login/security endpoints)?
  • Always return Retry-After. A bare 429 is incomplete — tell well-behaved clients when to retry (Retry-After) and how much budget is left (X-RateLimit-Remaining), or they hammer you in tight retry loops.
  • Used by — GitHub (5000 req/hr), Stripe, Twitter/X for API quotas; login/OTP endpoints for brute-force defense; Cloudflare/Akamai at the edge for DDoS mitigation.

Key takeaway

Rate limiting makes a per-request allow/reject decision in microseconds to cap each client. The token bucket — capacity B, refill r/sec — sustains an average rate while absorbing a controlled burst, and returns 429 + Retry-After when the budget runs out. At scale, keep the counter in an atomic, shared store like Redis so concurrent nodes can't overshoot the limit.
