Retries, Timeouts & Backoff — The Request Timeline

How a caller survives a flaky downstream: bound each call with a timeout, retry only transient errors, and space the retries with exponential backoff + jitter — so a small blip doesn't turn into a retry storm.

Real-world analogy — calling a friend on a busy line

You call a friend and the line is busy. Timeout decides how long you let it ring before hanging up — you don't hold forever. Retry is hanging up and dialling again, in case it was just a glitch. But if you redial instantly over and over, you (and everyone else calling) jam the line worse. Smarter: wait 1s, then 2s, then 4s between tries (exponential backoff). And add a little random wait — jitter — so the whole town doesn't redial at the exact same instant (that synchronized rush is the thundering herd). Backoff + jitter keeps retries calm and spread out, giving the line room to clear.

Retries with NO backoff / jitter

on fail → retry immediately, forever

A brief slowdown makes every client time out and re-fire at the same instant. Load spikes to 3–4× right when the downstream is already struggling — the classic retry storm / thundering herd. Synchronized bursts pile up:

Result: a minor blip cascades into a full outage.

Exponential backoff + full jitter

wait = random(0, base · 2^n), cap attempts

Each client waits a longer, randomized gap before retrying, so the same retries smear out over time instead of hammering at once. Peak load stays low and the downstream gets room to recover:

Result: transient blips absorbed, no amplification.

Things to know in the interview

  • Timeout off the p99, not a guess — set timeout a bit above the downstream's p99 latency (p99 ~20ms → timeout ~100–200ms), never the default 30s. Too tight → healthy calls fail; too loose → threads leak on a hung dependency.
  • Retry only transient errors — timeout, connection reset, 503, 429. Never retry 400/401/403/404: they fail deterministically, so retrying just adds load.
  • "Backoff" alone is incomplete — say "exponential backoff with jitter". Pure exponential still lets clients retry in lock-step at 100ms, 200ms… Full jitter random(0, base·2^n) is the AWS-recommended fix.
  • Cap attempts + overall deadline — e.g. 3 attempts total. Infinite retry DDoSes a truly-down service with your own traffic.
  • Idempotency required — a timeout doesn't mean the work didn't happen; the response may just be lost. Blindly retrying "charge card" double-charges. Use idempotency keys.
  • Don't nest retries — 3× at layer A times 3× at layer B = 9× load. Centralize retries at one layer or use a retry budget (e.g. retries ≤ 10% of requests, like gRPC/Envoy).
  • Pair with a circuit breaker — when failures are sustained (not transient), open the circuit and fail-fast instead of retrying, so the downstream can recover.

Key takeaway

Timeout bounds one slow call, retry re-attempts transient failures, and exponential backoff + jitter spreads those retries out so a struggling service isn't hammered. Always cap the attempts, retry only idempotent calls on retryable errors, and back it with a circuit breaker — otherwise retries amplify a small blip into a cascading outage.
