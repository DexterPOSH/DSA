Quorum — R + W > N

How a leaderless datastore with N replicas guarantees a read sees the latest write — by forcing the write set and the read set to overlap on at least one replica, without ever waiting for all N nodes.

Real-world analogy — the group-project friends

You and N = 5 friends each keep the latest copy of a group project on your own laptop. An update only counts as "saved" once at least W = 3 friends have copied it. When you want the newest version, you don't ask one friend — you ask any R = 3 friends and take the highest version number among them. Since R + W = 6 > 5 = N, the 3 who saved and the 3 you ask cannot all be different people — at least one friend is in both groups, and that friend hands you the latest copy. Drop to W = 2, R = 2 (4 < 5) and the two groups might never touch — you could read a stale version.

R + W > N  (2 + 2 > 3)

overlap = R + W − N = 1 ≥ 1

Write lands on R1, R2 (version v7). Read hits R2, R3. They share R2, which carries v7 — so the read always sees the latest value.

Result: strong-ish consistency — latest write is always visible.

R + W ≤ N  (1 + 1 ≤ 3)

overlap = R + W − N = −1 → can be 0

Write lands only on R1 (v7). Read hits only R3 (still v6). The sets never touch — the read returns a stale value.

Result: eventual consistency — faster, but stale reads possible.

Things to know in the interview

  • Why the overlap (pigeonhole)? Write touches W nodes, read touches R nodes. If R + W > N, two subsets can't fit inside N nodes without sharing — at least R + W − N replicas are common. For N=3, W=2, R=2 that's ≥ 1.
  • Don't wait for all N. With N=3, W=2, if two replicas ack in ~5ms and the third is slow (~40ms), the write returns at 5ms — one slow/GC-paused node is ignored, keeping tail latency low.
  • Read picks the winner by version — highest timestamp or dominating vector clock among the R responses. Quorum decides how many nodes; it does not resolve concurrent-write conflicts (that's LWW / vector clocks).
  • Read repair — if the R responses disagree, the coordinator pushes the latest value to the stale replicas in the background; it does not block the read.
  • Tuning knobs. Read-heavy: R=1, W=N (fast reads, fragile writes). Write-heavy: W=1, R=N. Balanced default: majority W=R=N/2+1 (N=3→2, N=5→3).
  • W = N is NOT "best". It blocks every write if a single replica is down — availability collapses. Production prefers majority, not full N.
  • Used by — Amazon Dynamo, Apache Cassandra (incl. LOCAL_QUORUM), Riak, Voldemort. Raft/Paxos use the same idea: any two majorities of N/2+1 always overlap.

Key takeaway

Confirm each write on W replicas and each read from R replicas; set R + W > N so the write set and read set always share at least one replica. That shared node carries the newest version, so a read is guaranteed to see the latest write — all without ever waiting for every replica.
