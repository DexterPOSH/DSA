Consensus — Raft / Paxos

How a cluster of unreliable nodes agrees on one ordered sequence of operations — surviving crashes and message loss — using leader election + majority quorum + monotonic terms. We walk a 5-node Raft cluster through electing a leader, replicating a write to a majority, and recovering from a leader crash.

Real-world analogy — the WhatsApp group plan

5 dosto ka group Sunday ka plan decide kar raha hai — movie ya trekking — par network flaky hai (kisi ka message late, koi offline). Rule banta hai: ek leader hoga aur har plan-change usi ke through propose hoga. Agar leader ka phone band ho gaya, baaki log timeout ke baad naya leader chun lete hain aur plan chalta rehta hai. Yahi consensus hai.

2-Phase Commit (2PC) — blocking

coordinator → all participants must vote YES

Needs every participant to agree. If the coordinator crashes mid-commit, participants are stuck uncertain — holding locks, unable to proceed or abort. A single failure can block the whole system. 2PC is for atomic commit across resources, not availability under failure.

Consensus (Raft / Paxos) — non-blocking

leader → MAJORITY acks = committed

Only needs a majority, not everyone. If the leader crashes, survivors time out and elect a new one — the cluster keeps going. Tolerates f = 2 of 5 down. Higher term always wins, so split-brain self-heals. This is replicated agreement that stays available.

Things to know in the interview

  • Quorum overlap is the whole trick — any two majorities of floor(N/2)+1 share at least one node, so two conflicting values can never both be committed. Say this and the interviewer smiles.
  • Odd numbers only — 4 nodes tolerate the same 1 failure as 3 (quorum 3 vs 2), just at extra cost. Pick 3, 5, or 7. 2f+1 nodes → f failures.
  • Term = logical clock — a monotonically increasing integer; at most one leader per term. See a higher term → step down to follower immediately. This kills stale leaders and split-brain.
  • Committed ≠ visible instantly — an entry is committed once durable on a majority, but the client sees it only after the state machine applies it, strictly in log order.
  • Raft vs Paxos — same safety, same 2f+1 math. Raft = strong leader + explicit terms, easier to grasp. Multi-Paxos amortizes phase 1 with a stable leader, ending up Raft-like in steady state. Default to Raft in interviews.
  • CP, not AP — partition a leader onto the minority side and it stops accepting writes (no quorum). Consensus trades availability for consistency.
  • Cross-region hurts — a commit needs one round-trip leader→majority→leader: ~1–5 ms same-region but 50–100+ ms across datacenters. So big systems shard and run one Raft group per shard (Spanner, CockroachDB, TiKV).
  • Used by — etcd, Consul (Raft); ZooKeeper (ZAB); Kubernetes control plane (etcd); Spanner (Paxos per shard).

Key takeaway

Consensus = elect one leader, funnel every write through it, and treat a write as committed only once a majority has durably stored it. Because any two majorities overlap, the cluster can never commit two conflicting values; and because a higher term always wins, a crashed leader is cleanly replaced — giving you a single, consistent, fault-tolerant source of truth at the cost of one round-trip per write.
