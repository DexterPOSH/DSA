Distributed Locks — Lease, Crash & the Fencing Token

How multiple machines agree that only one of them may touch a shared resource at a time — using an atomic SET NX PX lock, a self-expiring lease (TTL) so a crash can't deadlock forever, and a fencing token that stays safe even when a paused client wrongly thinks it still holds the lock.

Real-world analogy — the single conference-room key

One office, one conference room, and a single physical key at reception. Any team that wants the room must first grab that key — everyone else waits, no matter how senior. Meeting done, key goes back, next team's turn. The twist: if a team walks out mid-meeting still holding the key (a process crash), the room is locked forever. So the key is stamped "valid for 30 seconds" — a lease. After it expires the key auto-returns to reception, so the system heals itself even when someone vanishes.

Lease alone (no fencing)

hold lock → assume single writer → write

A holder hits a long GC pause or network stall. Its 30s lease silently expires; the Lock Manager hands the lock to Client B. The paused client wakes up still believing it holds the lock and writes — now 2 writers clobber the resource. More Redis nodes (Redlock) does not fix this.

Lease + monotonic fencing token

every grant carries token++ → resource keeps max(token)

A's grant carried token=33; B's grant carried token=34. The resource has already seen 34, so A's stale 33 write is rejected. Correctness no longer depends on clocks or timing — the resource itself enforces single-writer order.

Things to know in the interview

  • Acquire must be atomic — SET key token NX PX 30000 in one round-trip. NX sets only if the key is absent (lock is free); PX 30000 attaches a 30s TTL. A same-datacenter Redis acquire is ~sub-ms to 1-2 ms, so a service can do thousands of lock ops/sec.
  • Release must be compare-and-delete — never plain GET then DEL. Between the two the lease can expire and pass to someone else, and you'd delete their lock. Use a Lua script: if GET==token then DEL.
  • TTL is mandatory — without a lease, a crashed holder deadlocks the resource forever. Too short → a slow client's lease expires mid-work; too long → slow crash recovery. A watchdog thread renews (e.g. every 10s on a 30s TTL) for long jobs.
  • Redis (AP) vs ZooKeeper/etcd (CP) — a single Redis node is a SPOF: async replication leaves a tiny window where two clients hold the lock. CP stores give linearizable locks via consensus (ZAB/Raft) at higher latency (single-digit to low double-digit ms). Money/inventory → CP; pure efficiency → Redis is fine.
  • Fencing tokens beat "more nodes" — Redlock does not guarantee safety under GC pauses or clock skew. The real fix is a monotonically increasing token the resource checks.
  • Used by — Redisson (Java) on Redis, Google Chubby & ZooKeeper for leader election, HashiCorp Consul/etcd sessions, Kubernetes Lease objects (its leader election runs on an etcd lease).

Key takeaway

A distributed lock gives mutual exclusion across machines via an atomic acquire and a token-checked release — but a lease/TTL is what keeps a crash from deadlocking forever, and a monotonic fencing token validated by the resource is what keeps a paused-but-stale holder from corrupting data. Lock ≠ transaction: it only buys you "one writer at a time."
