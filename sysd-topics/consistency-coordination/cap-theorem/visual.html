CAP Theorem — The Partition Decision

During a network partition, a distributed system serving one shared dataset can keep only one of two promises: Consistency (every read sees the latest write) or Availability (every request gets a non-error response). It cannot keep both — so it must choose CP or AP.

Real-world analogy — two shops, one price list

Two shops — one in Mumbai, one in Delhi — sell from one shared price list. Normally Mumbai phones Delhi the moment a price changes, so both lists match. One day the phone line between them is cut (the network partition). A customer in Delhi asks for a price. Delhi has only two choices: (CP) refuse to serve — "I can't confirm Mumbai's latest price, so I won't risk a wrong one" — or (AP) serve from its possibly-stale local list anyway. When the line is restored, the shops reconcile. The key insight: while the line is alive there is no tradeoff — you get both. CAP's tension fires only during the partition.

CP — Consistency wins

minority can't reach W=2 → reject

The lone node N3 can't confirm a quorum, so it blocks or errors the request rather than risk serving stale data. The client waits out the quorum timeout and fails after about 500ms. Consistency is safe; that side's availability is gone.

Use when stale/wrong is unacceptable: money, inventory, locks, leader election.
Examples: etcd, ZooKeeper, Consul, Spanner, HBase.

AP — Availability wins

N3 accepts locally, reconcile later

The lone node N3 writes to its own copy and answers in ~5ms. Replicas now diverge; after the partition heals they reconcile (last-write-wins, version vectors, CRDTs). Availability is safe; consistency is only eventual.

Use when downtime is unacceptable and staleness is tolerable: carts, feeds, sessions, counters.
Examples: DynamoDB, Cassandra, Riak, CouchDB.

Things to know in the interview

  • "Pick any 2 of C/A/P" is the wrong framing. You can't opt out of P — networks drop packets in any real distributed system. So the practical choice is only "during a partition, give up C or A?" Real systems are effectively CP or AP; a true CA system exists only on a single non-distributed node.
  • No partition → no tradeoff. When the network is healthy a system can deliver strong consistency and high availability at once. CAP's tension is active only inside the partition window.
  • PACELC extends CAP. If Partition → A vs C; Else → Latency vs Consistency. Even with no partition, strong consistency needs cross-node coordination that raises latency. DynamoDB = PA/EL; a strongly-consistent store = PC/EC.
  • Quorum math. Choose W + R > N so every read overlaps the latest write — e.g. N=3, W=2, R=2 (CP). Set W=1, R=1 (2 ≤ 3) and writes always succeed but reads can be stale (AP-leaning).
  • CAP's "C" ≠ ACID's "C". CAP's C is linearizability across replicas; ACID's C is preserving transaction invariants (e.g. balance ≥ 0). Don't conflate them.
  • AP isn't "no consistency." It means no strong guarantee during a partition; read-repair, anti-entropy and CRDTs converge replicas after heal — that's eventual consistency.

Key takeaway

Partition tolerance is a given, not a choice — so the real CAP decision is what to sacrifice during a partition: a CP system rejects the minority write (waits the quorum timeout, ~500ms, then fails) to stay correct, while an AP system accepts it locally in ~5ms and reconciles after heal. With no partition you keep both — and PACELC reminds you the everyday tradeoff is then latency-vs-consistency.
