Sharding & Partitioning — Splitting One Big Dataset

When a single machine runs out of write throughput or storage, you break the dataset into independent chunks (shards) on separate nodes. A router reads the shard key and sends each request to exactly one shard — so the system scales horizontally instead of hitting one node's ceiling.

Real-world analogy — one giant library becomes branches

A city has one giant library: 10 crore books, one building, one entry gate. As books grow, the building runs out of space and the single gate jams with queues — one librarian can't cope. That's a single overloaded database. The city splits it into branches: authors A–G in branch A, H–N in branch B, O–Z in branch C — each with its own building, librarian and gate (this is range partitioning). But if "Harry Potter" gets super popular, its branch is mobbed while others sit idle — a hot shard. Fix: pick the branch by hashing the title so popular and quiet books scatter evenly. Trade-off: now "all A-author books" live in many branches, so that one query must visit them all.

Replication

SAME data, copied to many nodes

Every node holds a full copy. Scales reads & gives availability (a replica can take over). It does not help write throughput or storage — every node still stores everything.

Sharding

DIFFERENT data, split across nodes

Each node holds a distinct slice. Scales writes + storage — 10 shards ≈ 10× the write throughput and capacity. The classic interview trap: these are complementary, real systems shard AND replicate each shard.

Things to know in the interview

  • Shard key is everything — pick a high-cardinality field with even access (user_id, tenant_id). A low-cardinality key like country (80% traffic from one country) leaves some shards hot and others idle. Wrong key = full re-design.
  • Range → fast BETWEEN/scan queries, but a monotonic key (auto-increment id, created_at) piles every new write on the latest shard → hotspot. Fix: hash it, or shard on a high-cardinality dimension.
  • Hash → uniform spread, but range queries scatter. Naive hash(key) % N remaps almost everything when N changes → use consistent hashing or a fixed count of logical partitions (e.g. 1024).
  • Directory → a key → shard lookup table; max flexibility & easy rebalance, but adds a hop and a SPOF.
  • Cross-shard = scatter-gather — a query not keyed on the shard key hits every shard and merges; slow and tail-latency heavy (more shards → higher chance one is slow). Push such reads to a denormalized index (e.g. Elasticsearch).
  • Resharding isn't free — live migration needs dual writes, backfill, cutover, verification. And a celebrity user can overload one shard even with even data → split that key or cache it.
  • Used by — Instagram/Notion (Postgres on user_id/workspace_id), Vitess (MySQL: YouTube, Slack), MongoDB/Cassandra/DynamoDB (partition key), Discord (Cassandra on channel_id).

Key takeaway

Sharding splits different data across nodes (not copies — that's replication) so writes and storage scale horizontally. A router uses the shard key + a strategy (range, hash, or directory) to send each request to one shard. The whole game is choosing a high-cardinality, evenly-accessed shard key so no shard goes hot — and keeping each logical operation on a single shard to avoid slow scatter-gather.
