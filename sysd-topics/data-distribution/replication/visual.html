Replication — Leader, Followers & the Replication Log

How a system keeps multiple copies of the same data on different nodes for read scaling, availability, and fault tolerance — and the central pain it creates: copies are easy to make, hard to keep consistent. Watch a write flow through single-leader, then multi-leader and leaderless (N=3, W=2, R=2) topologies.

Real-world analogy — the famous book in a library

One library has a hugely popular book. With a single copy, only one person reads at a time, and if it's lost or torn — the book is gone (total data loss). So the library prints 5 copies across branches: now 5 people read at once (read scale), a burnt branch doesn't lose the book (fault tolerance), and Delhi reads from Delhi (locality / low latency). The new headache: when the author ships a new edition (a write), all 5 copies must be updated. Until they sync, different branches hand readers different versions — that's replication lag, and the heart of the topic.

Asynchronous replication

leader commits locally → ack NOW → followers catch up later

Ack is sub-ms fast and the leader never blocks on a slow follower — great latency & availability. But if the leader crashes before the log reaches a follower, those un-replicated, already-acked writes are lost. Followers also serve stale reads (replication lag, few ms → several seconds).

Synchronous replication

leader waits for ≥1 follower to confirm → THEN ack

No data loss on leader failure — the write is already durable on another node. Cost: +1 round-trip on every write (a 5 ms-away follower adds 5 ms each time), and if that follower is slow/down, writes stall — durability bought with availability (a practical taste of CAP). Middle ground: semi-sync — one follower sync, the rest async.

Things to know in the interview

  • Replication ≠ sharding. Replication = same data in multiple full copies (availability + read scale). Sharding = different data on different nodes (write/storage scale). Real systems usually do both: partition, then replicate each partition.
  • Async data loss is real. A client can get OK, then the leader crashes before the log ships — those writes vanish permanently. "We replicate, so we never lose data" is false unless sync/quorum-durable.
  • Replication lag side-effects. Stale follower reads cause the classic bug: update profile, reload, see old data. Fixes: read-your-writes (read your own writes from the leader), monotonic reads, or sticky routing.
  • Leaderless quorum. Write to N replicas, wait for W acks; read from R. If W + R > N the read & write sets overlap → you touch ≥1 up-to-date replica. e.g. N=3, W=2, R=2. But this is not linearizability — concurrent writes, sloppy quorums & read-repair timing still leak stale/conflicting values.
  • Failover isn't free. Detection timeout + new-leader election + client reroute all take time. With async, the old leader returning can cause split-brain (two leaders) and silently discard un-replicated writes — you need fencing / single-writer guarantees.
  • Used by — MySQL/PostgreSQL read replicas (binlog/WAL), MongoDB replica sets (Raft-like election), Kafka partition replication (ISR), Cassandra/DynamoDB/Riak (leaderless quorums).

Key takeaway

Replication keeps multiple copies of the same data so reads scale and a node (or whole data center) can die without data loss. The hard part is keeping copies consistent: async is fast but risks losing acked writes and serving stale reads, sync is durable but adds latency and can stall writes, and leaderless quorums tune consistency via W + R > N. Pick the topology and sync mode by what you can afford to lose — and always address failover explicitly.
