Consistent Hashing — The Hash Ring

How distributed caches and databases spread keys across nodes so that adding or removing one node moves only a tiny slice of keys — instead of reshuffling almost everything like plain hash(key) % N.

Real-world analogy — the round dining table

Picture a big round dining table. Dishes (your keys) are placed at spots all around the rim. Waiters (your nodes) also stand at fixed spots around the table. The rule: each dish is served by the first waiter you reach walking clockwise from that dish. If a waiter leaves, only the dishes between them and the previous waiter need a new server — every other dish keeps the same waiter. When a new waiter squeezes in, they only take over the dishes sitting just clockwise-behind them. No full re-seating of the whole table.

Naive  hash(key) % N

node = hash(key) % N

Looks simple — until N changes. Go from N=4 to N=5 and the modulus shifts for almost every key, so ~80%+ of keys land on a different node at once.

Result: a cache stampede — nearly every lookup misses, all traffic slams the database simultaneously.

Consistent hashing (the ring)

node = first clockwise on ring

Adding the 5th node only steals the keys in one arc — its predecessor's slice. With 1000 keys over 4 nodes, only about ~K/N = 200 keys move; the other ~800 never notice.

Result: graceful scaling. Each add/remove disturbs roughly K/N keys, not K.

Things to know in the interview

  • Why a ring? Wrapping the hash space into a circle means "the largest key" wraps back to "the smallest node," so there is always a next node clockwise — no edge cases at the boundary.
  • Virtual nodes (vnodes) — one physical server is hashed to many ring points (e.g. 100-200). This evens out the arc sizes so no single node gets a giant slice (a hot spot), and lets you weight bigger servers by giving them more vnodes.
  • Lookup cost — store node positions in a sorted structure and binary-search for the first position ≥ hash(key): O(log V) where V = total vnodes.
  • Replication — for redundancy, a key is often stored on the next R distinct physical nodes clockwise, not just the first.
  • Removal — a node's keys all flow to the next node clockwise; only that one neighbor absorbs the load, and only that arc moves.
  • Used by — Amazon DynamoDB, Apache Cassandra (token ring), Riak, and distributed caches like memcached rings / Redis Cluster (slot variant).

Key takeaway

Hash both keys and nodes onto the same circular space and assign each key to the first node clockwise. Then a node joining or leaving only reshuffles the ~K/N keys in one arc instead of nearly all of them — and virtual nodes keep those arcs evenly sized so no server becomes a hot spot.
