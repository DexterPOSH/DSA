Design a Notification System — The Event Pipeline

How one trigger event from a sender service flows through a decoupled, queue-based pipeline and fans out to millions of users across Push, SMS, and Email — reliably and near-real-time. Scale anchor: 10M DAU, 50M notifications/day, ~580 QPS avg → ~3K QPS peak.

Real-world analogy — the restaurant kitchen

A sender service is a waiter who shouts an order. Instead of running it to the stove themselves, they clip the ticket onto the order rail (the Kafka queue) and immediately go back to the floor — that is the 202 Accepted. Line cooks (the channel workers) pull tickets off the rail at their own pace: each one checks the recipe card (preferences + DND), confirms the table hasn't already been served too much (rate limit), plates the dish (template render), and hands it to the right station — grill, fryer, or salad (Push, SMS, Email providers). If the fryer is jammed, tickets just wait on the rail; nobody at the front desk is blocked, and no order is thrown away.

Things to know in the interview

  • Why a queue at all? Calling providers synchronously is the anti-pattern — one slow provider blocks the sender and a spike topples the system. Kafka decouples producer from consumer: spikes grow queue depth instead of dropping, and failed messages can be re-enqueued.
  • Priority isolation — separate topics for transactional (OTP, password reset) vs bulk (marketing) so an OTP never gets stuck behind a campaign. Partition by user_id for soft per-user ordering.
  • At-least-once + dedup — exactly-once is impractical in distributed delivery, so accept duplicates and kill them: SETNX dedup_key in Redis (with TTL) before the provider call; already set → skip.
  • Failure handling — provider errors → retry with exponential backoff; poison messages → Dead Letter Queue so one bad event can't block a whole consumer.
  • Provider resilience — wrap each adapter in a circuit breaker; if APNs error rate spikes, open the circuit and hold traffic in the queue instead of cascading. Multi-provider failover (SES → SendGrid).
  • Two stores, two access patterns — preferences in SQL (low write, strong consistency, compliance), notification log in Cassandra (write-heavy 50M/day, append-only, partition by user + cluster by time).

Key takeaway

The whole design hinges on decoupling via a message queue: the API returns 202 instantly and drops the event on Kafka, so traffic spikes and provider outages absorb into queue depth rather than dropped notifications. Channel workers then independently apply preferences/DND, rate limits, templating, and dedup before calling third-party providers — with at-least-once delivery + idempotency, retries, DLQ, and circuit breakers turning an unreliable fan-out into a reliable one.
