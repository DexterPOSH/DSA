Design a Web Crawler — The Crawl Loop

How a distributed crawler fetches ~1 billion pages/month (~400 pages/sec, ~1.6 Gbps inbound) by sending each URL around a loop: Frontier → Fetch → Store → Parse → Dedup → back to Frontier — staying polite to every host and never crawling the same page twice.

Real-world analogy — exploring a city with a shared to-do list

Imagine a team of explorers mapping a giant city. They share one to-do list of streets to visit (the URL Frontier). An explorer takes the next street off the list, walks it (fetch the page), writes down what they saw in a shared notebook (content store), and notes every new street sign they spotted (extracted links). Before adding a new street to the list, they check a "streets we've already mapped" ledger (dedup / seen-set) so nobody walks the same street twice. And there's etiquette: don't send ten explorers down one tiny alley at once — knock politely, obey the alley's posted rules (robots.txt), and space out your visits. The list never empties on its own; it keeps growing from the signs you find.

Inside the URL Frontier — Mercator-style two-tier queues

The frontier is the crawler's brain. It juggles two conflicting goals at once: crawl important pages first (prioritization) and never hammer a single host (politeness). Two tiers of queues solve both.

Front queues → prioritization

One queue per priority level. A PageRank / freshness signal assigns each URL a priority; a biased router pulls high-priority URLs more often.

One queue per host. All URLs of a host land in the same queue, so one worker drains it serially → automatic per-host rate limiting. A heap tracks each host's next-ready time.

Things to know in the interview

  • Dedup at scale = Bloom filter + content hash. Billions of URLs won't fit an exact RAM set, so a Bloom filter (~10 bits/elem → ~1% false-positive) answers "definitely new" vs "maybe seen" in ~5 GB; only "maybe" hits the backing KV store. Separately, a SHA-256 content hash makes the blob store content-addressed, so mirror pages with the same bytes are stored once.
  • Politeness needs caching. Cache robots.txt per host (with TTL) and run a dedicated cached DNS resolver — millions of synchronous, uncached DNS lookups would throttle the whole fleet.
  • Storage & bandwidth dominate, not CPU: ~1.6 Gbps sustained inbound and ~1.2 PB/year compressed → fetchers scale horizontally and content lands in a blob store (S3 / HDFS).
  • Frontier is a hotspot / SPOF. Shard it by host (domain hash) so one host always maps to one shard — per-host serialization survives and throughput scales linearly.
  • Spider traps. Calendars and infinite generated URLs trap the crawler. Defend with per-host URL budgets, max crawl depth, URL-length limits, and blacklists.
  • NoSQL, not SQL. URL metadata is a simple key-based lookup/upsert by url_hash at huge write volume — wide-column stores (Cassandra / Bigtable) shard naturally; SQL joins & ACID are wasted here and become the bottleneck.

Key takeaway

A web crawler is a loop, not a pipeline: every fetched page feeds links back into the frontier, so the design lives or dies on two guards — a two-tier frontier (front queues for priority, back queues for per-host politeness) and two-level dedup (Bloom filter on URLs, content hash on bytes). Get those right and the rest — fetchers, blob store, DNS/robots caches — just scale horizontally.
