Design Twitter / News Feed — The Fan-out Flow

How a single tweet reaches millions of home timelines fast. Twitter is read-heavy (~1M reads/sec peak vs ~12K writes/sec), so the home timeline is precomputed on write — except for celebrities, who break the model and get a hybrid push/pull treatment.

Real-world analogy — the newspaper round vs. the notice board

Think of two ways news reaches you. Push (fan-out on write): a paperboy delivers a fresh copy to every subscriber's doorstep the moment it's printed — when you wake up, your paper is already waiting (instant read). The cost is all the legwork at delivery time, multiplied by every subscriber. Pull (fan-out on read): instead, the news sits on a central notice board and you walk over and read it when you want — cheap to publish, but you do the work every single visit. A celebrity with 50 million "subscribers" can't have a paperboy run to all 50M doors for one tweet — that's a stampede. So their tweets stay on the notice board, and you merge them in only when you actually open your feed.

Fan-out on write (push)

on tweet: LPUSH into N follower lists
  • Reads are instant — feed is a ready-made Redis list.
  • Cost: write amplification — 1 tweet × N followers = N writes.
  • Perfect for the ~100:1 read:write ratio: make reads cheap.

Default for normal users. Async via Kafka so write latency stays low.

Fan-out on read (pull)

on read: query + merge followees' tweets
  • Writes are cheap — tweet stored once, nothing fanned out.
  • Cost: heavy compute every read — query & merge each time.
  • Great for high-fanout celebrities (avoid N-million writes).

Too slow as a default at 1M reads/sec.

The hybrid — Twitter's real answer

Normal users → push (precompute, fast reads). Celebrities above a threshold (e.g. >10K–1M followers) → do NOT push — one tweet would mean millions of Redis writes (a "thundering herd"). Their tweets stay put and are pulled and merged at read time into each follower's pushed timeline. This single switch kills the #1 bottleneck: celebrity fan-out / write amplification.

Things to know in the interview

  • Why async via Kafka? The write returns as soon as the tweet is persisted; fan-out runs in the background. Decoupling keeps write latency low and gives back-pressure + retries so tweets buffer (not lost) when workers lag.
  • Capped Redis list — timeline:{uid} holds only ~800 recent tweet_ids (not full content). 1M read QPS serves from this cache; content is batch-hydrated separately from the Tweet Cache.
  • Active vs inactive users — don't precompute feeds for users dormant for weeks. Push for active users only; lazily pull-compute for inactive ones to save Redis memory.
  • Eventual consistency — because push is async, a new tweet can show up 1–2s late in a follower's feed. That's an accepted tradeoff for high availability + low read latency (AP-leaning).
  • Snowflake IDs — tweet_ids are 64-bit time-sortable (timestamp + machine + sequence) so timelines merge in reverse-chronological order with no central ID bottleneck.
  • Cursor pagination — opaque cursor (last tweet_id/timestamp), not offset, so a constantly-changing feed doesn't produce duplicates or skips.

Key takeaway

Because Twitter is massively read-heavy, you precompute the home timeline on write (push fan-out via Kafka into capped Redis lists) so reads are a cheap cache lookup. But blind push explodes for celebrities, so you go hybrid: push for normal users, pull-and-merge celebrity tweets at read time — trading a touch of eventual consistency for availability and sub-200ms feeds.
