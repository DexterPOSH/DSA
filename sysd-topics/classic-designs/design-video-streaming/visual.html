Design YouTube / Video Streaming — End-to-End Flow

How a video travels from a creator's upload, through an async transcoding pipeline, into durable object storage, and back out to billions of viewers via a CDN with adaptive bitrate — a ~1000:1 read-heavy system serving ~3.5 Tbps of egress.

Real-world analogy — a film studio + global cinema chain

A creator hands over one master reel (the raw upload). The studio's lab doesn't make the audience wait — it sends the reel down a back-room processing line (the transcoding pipeline) that prints copies in many sizes: a tiny one for phones (240p), a big one for TVs (4K), all cut into short clips. Those prints are filed in a giant fireproof vault (the object store, 11-nines durable). When viewers worldwide want to watch, they don't queue at the vault — each city has a local cinema (the CDN edge) that keeps popular reels on hand. Your player even picks the print size that matches your seat — sorry, your network speed (adaptive bitrate).

Naive  synchronous + origin-served

upload → transcode inline → serve bytes from origin

App server takes the upload, transcodes in the request, and streams every byte itself. Transcoding takes minutes, so the upload call blocks; and serving ~3.5 Tbps from origin is physically impossible.

Result: uploads time out, the worker fleet melts, and the origin's bandwidth bill is a brick wall. The system cannot exist.

This design  async + CDN

direct-to-store upload · queue · CDN-served reads

Bytes bypass app servers (pre-signed URLs). Transcoding runs async behind a queue, scaling on lag. Reads are served from CDN edges, so origin only sees cache misses.

Result: uploads return instantly, processing autoscales, and ~99%+ of watch bytes never touch the origin.

Things to know in the interview

  • Bytes never flow through app servers — both upload (pre-signed multipart PUT) and download (CDN) go straight to/from object storage. App servers handle only the control plane: metadata, status, orchestration.
  • Transcoding is async + DAG-parallel — split the video into segments, fan-out transcode each segment × each resolution in parallel, then merge into HLS/DASH manifests. Needs idempotency + retries so a crashed worker reprocesses without duplicate output.
  • CDN is non-negotiable — ~3.5 Tbps egress can't come from origin. Store 2-10s chunks; popular content stays hot at the edge, cold/tail videos pull from origin on first miss (origin shielding mitigates the miss storm on a new viral video).
  • Adaptive bitrate (ABR) — player fetches a master manifest, then picks a rendition per network conditions (480p on slow, 1080p on fast), switching mid-stream to minimize buffering.
  • View counts are eventually consistent — writing the DB on every view kills throughput. Fire view events to a queue, batch/stream-aggregate, materialize periodically. Acceptable because availability > strong consistency here.
  • Data model — videos in NoSQL (partition by video_id, point lookups), accounts/billing in SQL, and actual bytes only in the object store — the DB stores paths/URLs, never the blob.

Key takeaway

Separate the write path from the read path: uploads land directly in durable object storage and trigger an async, queue-driven transcoding pipeline that produces multi-resolution chunks; reads are served from a CDN with adaptive bitrate so the origin only handles cache misses. App servers stay on the lean control plane, and availability beats strong consistency — so a stream never stalls just because a view count is a few seconds behind.
