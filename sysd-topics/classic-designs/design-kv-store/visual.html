Distributed Key-Value Store — The Quorum Read/Write Path

A peer-to-peer, Dynamo-style store: any node can coordinate, keys live on the next N nodes clockwise on a consistent-hash ring, and a tunable R/W quorum decides how many replicas must answer before the client hears "done". Watch one put and one get travel the ring — including a node failure.

Real-world analogy — sending a memo to a committee

You hand a memo to any staff member you bump into — they become the coordinator for your request. The memo is filed in 3 identical cabinets (the replicas, N=3) kept in different rooms for safety. You don't wait for all three clerks to confirm; you leave as soon as any 2 of them shout "filed!" (the write quorum, W=2). Later, to read it back, you ask all three and accept the answer once 2 reply (R=2). Because the 2 who confirmed your write and the 2 who answer your read must share at least one clerk (2 + 2 > 3), that overlapping clerk is guaranteed to be holding the newest copy — so you never read stale. If a clerk is out sick, a stand-in takes the memo with a sticky note (hinted handoff) and passes it along when they're back.

Tuned for consistency   N=3, R=2, W=2

R + W = 2 + 2 = 4 > N = 3 ✓

Write and read quorums always share ≥1 replica, so a read is guaranteed to touch the freshest copy — read-your-writes. Latency = the 2nd-fastest replica, so one slow/dead node never blocks you. The repo's default.

Tuned for speed   W=1, R=1

R + W = 1 + 1 = 2 > N = 3 ✗

Ack after a single replica — lowest possible latency and "always writable", but quorums may not overlap, so a read can return stale data until anti-entropy catches up. This is the AP, eventually-consistent end of the dial.

Things to know in the interview

  • Why peer-to-peer? No single master = no single bottleneck or SPOF. Any node can coordinate; a smart client hashes the key itself and hits a replica directly, saving a hop.
  • Hinted handoff — if a replica is down at write time, the coordinator stores the write on a healthy stand-in with a hint. Writes stay available (the W quorum still completes); the hint is replayed when the node returns.
  • Read-repair — when a read finds replicas disagreeing on version, the coordinator pushes the newest value to the stale replicas in the background, so divergence self-heals on the hot path.
  • Conflicts — concurrent writes are tracked with vector clocks; causally-ordered ones auto-resolve, truly-concurrent ones are returned to the client (or resolved last-write-wins by timestamp).
  • Anti-entropy — replicas periodically compare Merkle trees and stream only the divergent key ranges, reconciling without scanning all data.
  • Storage engine — each node persists with an LSM-tree (commit log → memtable → SSTables + compaction), tuned for high write throughput; Bloom filters skip absent keys on reads.
  • Used by — Amazon DynamoDB, Apache Cassandra, Riak — all consistent-hash + quorum + gossip membership.

Key takeaway

A distributed KV store places each key on the next N nodes clockwise on a consistent-hash ring, then lets you dial consistency with a quorum: acknowledge a write after W replicas and a read after R. Pick R + W > N (e.g. N=3, R=2, W=2) and the two quorums always overlap, guaranteeing fresh reads — while hinted handoff and read-repair keep the system writable and self-healing when nodes fail.
