Design a Rate Limiter — The Token Bucket

How a server-side limiter caps a client to 100 requests per minute by letting each request allow_request(key, rule) spend a token from a shared, atomically-updated bucket in Redis — returning 429 Too Many Requests when the bucket runs dry.

Real-world analogy — the arcade token machine

Think of an arcade where a machine slowly drops fresh tokens into a bucket at a steady rate — say ~1.67 tokens every second (that's 100 per minute). The bucket can hold at most 100 tokens; once full, extra drops spill out and are wasted. Every time you want to play a game (send a request), you must spend one token. If the bucket has tokens, you play and the count drops by one. If you arrive after a quiet spell, the bucket is full, so you can fire off a quick burst of games back-to-back. But hammer it non-stop and the bucket empties — now you wait at the steady refill rate. That's exactly a token-bucket rate limiter: bursts allowed when you have credit, but a hard steady-state cap.

Fixed window counter

INCR key; if first → EXPIRE 60

Dead simple and fast, but a client can fire 100 in the last second of one window and 100 in the first second of the next — 200 requests in a ~1s span despite a "100/min" limit. That's the boundary burst bug.

Two windows, counted separately, let the burst slip through the seam.

Token bucket (recommended)

refill by elapsed·r; if tokens≥1 spend 1

Tracks just 2 values (token count + last-refill timestamp), allows healthy bursts up to b, but enforces the steady rate r across every boundary. With ~58K avg QPS and ~175K peak, that tiny per-key state is what keeps Redis cheap.

The cap travels with time — no seam to exploit. Stripe / AWS use this.

Things to know in the interview

  • Atomicity beats the race. Naive "read → check → increment" lets two gateway boxes both read the same count and both allow, busting the limit. Make it one atomic op: INCR (fixed window) or a Redis Lua script (token bucket: refill + check + decrement server-side, in one shot).
  • Fail-open vs fail-closed. If Redis is down, most systems fail-open (allow the request) so the main service stays up — availability over correctness. For fraud / login limits, fail-closed can be the right call.
  • Distributed coordination. Centralized Redis = accurate but one extra network hop; local + periodic sync (sloppy counters) = low latency but a little over-limit slack. High-scale systems usually accept the slack.
  • Hot key. A viral key hammers one Redis shard. Split it into key:bucket_0..N sub-counters and sum, or consistent-hash to spread the load.
  • Storage is cheap, latency isn't. ~10M active keys × ~50 bytes ≈ 500 MB — fits one Redis instance. Counters are ephemeral (TTL = window), so the real pressure is QPS and sub-millisecond decisions on the hot path.
  • Client signal. On reject return 429 with Retry-After, plus X-RateLimit-Limit / Remaining / Reset headers so clients can back off cleanly.

Key takeaway

Put the limiter server-side in the gateway, store each client's token bucket (count + last-refill ts) in an in-memory store like Redis, and make the refill-check-decrement a single atomic operation (Lua script). Tokens give you burst tolerance with a hard steady-state cap; 429 + Retry-After tells the client to slow down — and when Redis dies you usually fail-open so the product keeps serving.
