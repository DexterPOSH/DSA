Design TinyURL — The Request Flow

How a URL shortener turns shorten(longUrl) into tinyurl.com/abc1234, then serves ~100K redirects/sec back to the original link — watch a single request travel through the load balancer, cache, database, and key-generation service.

Real-world analogy — the coat check

A URL shortener is a giant coat check. You hand over a bulky coat (your long URL) and get back a tiny numbered ticket (the short key abc1234). The ticket is small, unguessable, and unique. Later you (or anyone you give the ticket to) walk up, present the ticket, and the attendant instantly fetches your exact coat — a redirect to the original URL. The clever part: most people keep showing the same few tickets, so the attendant keeps those coats on a nearby rack (the cache) instead of walking to the deep storage room (the database) every time.

Key gen by hashing

key = base62(md5(longUrl))[0:7]

Hash the long URL, take the first 7 base62 chars. Simple, but two different URLs can hash to the same prefix — a collision. Every insert must check-and-retry (rehash / append), adding extra DB reads on the write path.

Key Generation Service (KGS)

key = pop(pre-generated unused pool)

A background service mints base62 keys offline into an "unused" table. App servers just grab one and mark it "used". Collisions are structurally impossible, and key creation is off the latency-critical path entirely.

Things to know in the interview

  • 301 vs 302 — 301 Permanent lets browsers/proxies cache the redirect, so later clicks never reach you (less load, but you lose click analytics). 302 Found routes every click through your server — pick it if you need analytics or a kill-switch.
  • Why counter + base62 over hashing? A globally-unique counter base62-encoded guarantees uniqueness with no collision check. Distribute the counter via ZooKeeper ranges or Snowflake-style IDs so it isn't a single bottleneck.
  • Cache is the real scaling lever — reads are 100× writes. The 80/20 rule means ~20% of links drive ~80% of clicks, so an LRU cache gets a high hit ratio. Mappings are immutable, so cache invalidation is essentially free.
  • Sharding 110 TB — shard on hash(shortKey) (or a consistent-hash ring), not on a range of the first char which creates hot shards. Lookup is deterministic because the redirect already knows the key.
  • SQL vs NoSQL — it's a pure key-value lookup with no joins, billions of rows. A NoSQL / wide-column store (DynamoDB, Cassandra) gives built-in partitioning; eventual consistency is fine for redirects.
  • Hot key (viral link) — one link can overwhelm a single cache/shard node. Replicate that key across multiple cache nodes or push the redirect to a CDN edge.

Key takeaway

TinyURL is a read-heavy key-value lookup: generate a short, collision-free base62 key (KGS or counter), store the key → longUrl mapping in a sharded NoSQL store, and put a big LRU cache in front so the ~100K redirects/sec are served from memory in single-digit milliseconds while the database stays the durable source of truth.
