Design Uber / Nearby

How a ride-hailing system swallows a 250K writes/sec location firehose, answers nearby drivers in < 200–300 ms, and matches a rider to one driver without double-booking. Follow the packet through the pipeline.

Real-world analogy — the air-traffic control tower

Think of a busy airport control tower. Every plane (a driver) constantly radios its position every few seconds — that's a relentless stream of chatter the tower must absorb without choking. The tower doesn't scan the whole sky for every question; it divides airspace into sectors / grid squares (the geo-index) so "who's near runway 9?" only inspects a few nearby squares, not the planet. And when a gate is assigned to a plane, the controller claims it atomically — two planes can never be told to use the same gate at the same instant. Absorb the firehose, bucket by region, claim exactly once.

Naive  scan all drivers

WHERE dist(p, driver) < r // over 1M rows

No spatial index → every nearby query is an O(N) scan of all 1M drivers. And funnelling 250K writes/sec of per-row updates into one relational DB makes that the bottleneck too.

Result: nearby queries blow past the latency budget and the write firehose melts a single store.

Geo-index + sharding + Kafka

GEOSEARCH on per-city Redis · async persist

Geohash/H3 buckets turn nearby into an O(log N) lookup over a handful of cells. Per-city shards spread the firehose; Kafka decouples durable history so ingest stays thin.

Result: fast reads, horizontal writes, and the claim stays transactional in SQL — no double-booking.

Things to know in the interview

  • Geo-index choice — start with geohash + Redis GEOSEARCH (sorted set keyed by geohash score, O(log N) radius query). Watch the cell-boundary gotcha: query the center cell + 8 neighbours so a driver just over the edge isn't missed. Upgrade to H3 (Uber's actual choice — hexagons, equidistant neighbours) or a quadtree for dense-area skew.
  • Tame the 250K-QPS firehose — keep ingest thin & ack-only; overwrite latest location in Redis (no append), put a 10–15s TTL so dead drivers self-evict; fan durable history to Kafka for downstream consumers.
  • Geo-sharding — shard the location store by city/region. Mumbai drivers on the Mumbai shard; nearby queries resolve inside one shard since rider + driver are usually same-city.
  • No double-booking — the claim is the strongly consistent step: SETNX driver:lock:{id} or UPDATE ... WHERE status='AVAILABLE'. Only the winner offers; decline/timeout releases the lock → next candidate.
  • Async matching — return 202 Accepted {rideId, status:MATCHING}; deliver the result via push/poll. Don't block synchronously on search + lock + confirm.
  • Two stores, two patterns — ephemeral location → Redis (eventual, AP-leaning); ride/booking → SQL (ACID); raw trail (~788 TB/yr) → object/time-series store, never the hot path.

Key takeaway

Split the system by access pattern: absorb the write firehose with a thin, ack-only ingest that overwrites into a sharded in-memory geo-index (and streams history to Kafka), serve nearby as an O(log N) spatial lookup over a few cells, and make the match a single atomic claim in a transactional store. Reads can be eventual; the booking must be exactly-once.
