Design a Chat System (WhatsApp) — Message Flow

How a single text message travels from Alice on Chat Server A to Bob on Chat Server B — the routing via the Session Service, the store-and-forward path when Bob is offline, and how the three ticks (✓ ✓✓ ✓✓) get earned along the way.

Real-world analogy — the post office with mailboxes

Think of a giant post office. Alice hands her letter to clerk A (her Chat Server). The clerk immediately stamps it "received" and gives Alice a receipt — that's the single tick. To find Bob, the clerk checks the front desk directory (Session Service: "which clerk is Bob standing at?"). If Bob is at clerk B's window, the letter is handed straight over and Bob signs for it — double tick. If Bob has gone home, the letter is dropped into Bob's personal mailbox (his per-user inbox in the DB) and a courier knocks on his door (push notification). When Bob returns, the clerk empties his mailbox and delivers everything. No letter is ever lost.

Recipient online — real-time

A → persist → Session(B) → bus → B → push

Bob's socket is live on Server B. Server A persists, looks Bob up in the Session Service, forwards over the internal message bus, and B pushes instantly. Bob's delivered ack flows back → Alice sees double tick; on open → blue tick. Sub-second, no notification needed.

Recipient offline — store & forward

A → persist → Session(none) → mailbox + push

Session Service returns not connected. The message is parked as pending in Bob's per-user mailbox and a push notification is fired. On reconnect, Server B drains the mailbox and delivers — only then does the double tick appear. Durability core: nothing is lost.

Things to know in the interview

  • Why stateful chat servers? Sockets live in server memory, so load-balancing must be sticky. On crash, all its clients reconnect at once (a thundering herd) — mitigate with jittered exponential backoff + fast Session Service updates.
  • The Session Service is service-discovery for live users — a userId → chatServerId map in Redis with short TTLs. Without it, Server A has no idea which server holds Bob's socket.
  • at-least-once + idempotency — every message carries a client-generated clientMsgId (UUID). If an ack is missed and the message is re-sent, the client de-dups → effectively-once delivery.
  • Per-conversation ordering only — messages get a time-sortable Snowflake/ULID id; within one conversation partition they cluster sorted by time. No global total order needed (it would cost latency).
  • Availability over strong consistency — a message arriving slightly late or out-of-order is fine; a lost message is not. That's the CAP choice for chat.
  • Group fan-out — one message to a 1000-member group becomes N copies + N receipts. Push fan-out onto an async queue (Kafka); consider fan-out-on-read and aggregated receipts for huge groups.

Key takeaway

A chat system is fundamentally a routing + durability problem on top of persistent sockets: the Session Service answers "which server holds the recipient?", the Message Store guarantees the write is durable the instant it arrives (single tick), and the per-user mailbox turns "offline" from a failure into a normal store-and-forward case. Acks earn the ticks, clientMsgId kills duplicates, and the whole thing optimizes for availability and never-lost over global ordering.
