Blob / Object Storage — Token-in, Blob-out

How a system stores billions of immutable objects (blob + metadata + key) in a flat bucket, served over an HTTP API — and exactly what happens on the write (PUT) and read (GET) paths.

Real-world analogy — the giant cloakroom

Picture a huge cloakroom. You hand your bag (the object = data + metadata) over the counter and get back a token slip with a unique number (the key). You never learn which shelf, room, or how many copies hold your bag — that's the cloakroom's internal job. To get it back, you just present the token (GET key) and the whole bag comes out. Two truths fall out of this: you can't reach in and swap one shirt — you take the whole bag, repack, and re-deposit (objects are immutable); and the cloakroom can hold millions of bags without your experience changing (flat namespace, near-infinite scale).

Object storage

HTTP · flat namespace
  • Immutable objects, no partial writes
  • Near-infinite scale, tens-of-ms per GET
  • Media, backups, data lakes, ML data
  • S3 · GCS · Azure Blob · MinIO

Block storage

raw volume · disk blocks
  • Low-latency random read/write (sub-ms)
  • Partial in-place writes
  • Databases, transactional / OLTP
  • EBS · raw volumes

File storage

POSIX paths · hierarchy
  • Real folders + partial writes
  • Shared file system semantics
  • Shared mounts, lift-and-shift apps
  • NFS · EFS

Things to know in the interview

  • Immutable, not editable — to change one byte you re-PUT the whole object. Need frequent small in-place updates (counters, DB rows)? Wrong tool — that's the classic trap.
  • Durability ≠ Availability — durability (e.g. 11 nines = 99.999999999%) means data won't be lost; availability (e.g. 99.99%) means it's reachable right now. You can be highly durable yet briefly unavailable during an outage.
  • Flat namespace — a/b/c.png has no real a/b/ directory; it's just a key prefix. "Rename a folder" isn't O(1) — every object must be copy + delete.
  • Strong read-after-write — modern S3 (since Dec 2020) and GCS return the latest data on a GET right after a PUT. "S3 is eventually consistent" is outdated.
  • Replication vs erasure coding — 3 copies = simple, fast recovery, 3× cost; Reed-Solomon 10+4 = same durability at ~1.4× overhead but CPU + reconstruction cost. Hot/small → replicate; cold/large → erasure-code.
  • Big files = multipart — split into parts (≥5 MB), upload in parallel, retry only failed parts, then one "complete" call assembles them. A single giant PUT that retries from zero is the wrong answer.
  • Cost is more than per-GB — request counts (PUT/GET), egress, and archive (Glacier) retrieval fees all add up; "Glacier is cheap" ignores minutes-to-hours retrieval.

Key takeaway

Object storage trades low-latency random access and in-place edits for immutable objects in a flat, HTTP-addressed namespace that scales almost without limit. A PUT authenticates, chunks, and replicates/erasure-codes the blob across disks & AZs while recording a key → location entry in a separate metadata index — earning 11-nines durability; a GET looks up that index and streams the bytes back, fronted by a CDN for hot reads.
