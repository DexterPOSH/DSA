SQL vs NoSQL — One Write, Two Worlds

Follow a single write request as it travels through a relational SQL engine (schema-on-write, ACID, scale up) and a NoSQL cluster (schema-on-read, BASE, scale out) — then watch what each does when a network partition splits the cluster.

Real-world analogy — the government form vs the free-form diary

SQL = a government office form. Every column is pre-defined (Name, DOB, Aadhaar), every field has a fixed type, and if one mandatory field is blank or wrong-typed, the whole form is rejected at the counter. If two clerks edit the same record at once, a lock keeps it from corrupting. Reliable but rigid — a new column means changing the whole template.

NoSQL = a free-form diary. Each page (document) can hold whatever you want — one page just a name & phone, the next a name, address, 3 emails and a nested order list. Super flexible and you can rip in new pages fast (horizontal scale), but no central rulebook forces every page into the same shape — keeping things consistent is now your application's job.

SQL — schema-on-write · ACID · scale-up

CREATE TABLE users (id INT, email VARCHAR UNIQUE, age INT);
  • Validates on write — wrong type / missing field is rejected at the door. Data integrity guaranteed (FK, UNIQUE, NOT NULL).
  • ACID via WAL + locks/MVCC — atomic commits, no double-spend.
  • Scales vertically: bigger box. One tuned node ≈ single-digit-k → low-tens-of-k writes/sec; then read replicas / sharding (Vitess).
  • First-class ad-hoc JOINs & a query planner — declarative.

NoSQL — schema-on-read · BASE · scale-out

put(partition_key=user_id, doc={...}) // any shape
  • No write-time schema — app interprets shape on read. Flexible, but risks schema drift.
  • BASE (Basically Available, Soft state, Eventually consistent) — replicas converge over time; modern stores offer tunable consistency.
  • Scales horizontally: add nodes, partition by key → millions of ops/sec, near-linear.
  • Query-driven modeling — denormalize for your access pattern; cross-partition JOINs are slow/absent.

Things to know in the interview

  • "NoSQL means no schema" — wrong. The schema still exists; it's just implicit in app code (schema-on-read) instead of enforced by the DB. Better framing: schema-on-write vs schema-on-read.
  • "NoSQL always scales better" — wrong. A bad partition key creates hot partitions and throughput collapses; an unindexed cross-partition scan can be slower than a SQL JOIN. Scale isn't free — it costs query-driven modeling.
  • "NoSQL = no transactions" — outdated. MongoDB 4.0+ has multi-document ACID; DynamoDB has transactions & strongly-consistent reads; Cassandra tunes per-query (ONE/QUORUM/ALL).
  • CAP only bites during a partition. Normal operation can have both C and A. SQL (single/sync-replica) leans CP; Dynamo-style leans AP — but it's tunable now.
  • NewSQL is the middle path. Spanner, CockroachDB, TiDB, Vitess give horizontal scale + SQL + (often) strong consistency. SQL-vs-NoSQL is a spectrum, not a binary.
  • Real systems are polyglot. Postgres for orders + Redis for cache + Elasticsearch for search — pick per access pattern.

Key takeaway

SQL validates structure up front (schema-on-write) and guards data with ACID, scaling up — ideal for money, orders, complex JOINs and strong consistency. NoSQL skips write-time schema (schema-on-read), embraces BASE / tunable consistency and scales out to millions of ops/sec — ideal for huge write-heavy, flexible-schema workloads. It's a spectrum (NewSQL bridges both), and real systems mix them.
