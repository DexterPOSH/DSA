B-Tree vs LSM-Tree — Two Ways to Put Data on Disk

Both are sorted on-disk storage engines, but they make opposite bets. A B-Tree does the hard work at write time (find the right page, update it in-place → random I/O). An LSM-Tree does the hard work at read time plus background cleanup (buffer in RAM, append sequentially, merge later). Watch a single PUT, then a GET, flow through each.

Real-world analogy — the classroom register

You keep a class register where students must stay alphabetically sorted.

B-Tree way: one proper register, each page sorted. New student "Karan" arrives → you flip to the Ka… page and physically insert him there (a random write); if the page is full you split it. Lookups are lightning-fast — follow the index straight to the name — but every write costs you a seek to the exact spot.

LSM way: you never write into the register directly. A small desk notebook (memtable) takes new names instantly, no sorting tension. When it fills, you sort it once and lay down a fresh clean sheet (immutable SSTable) on a pile, then grab a new notebook. Writing is super fast — you only ever move forward. Catch: to find "Karan" you check the notebook, then every sheet newest→oldest, so a peon (compaction) keeps merging sheets in the background to kill duplicates and keep reads sane.

B-Tree — pay at write time

lookup: 3–4 page reads · write: in-place (random)

A large DB packs 2³² keys into just 3–4 levels (high fan-out), so a point lookup is ~4 page reads, sub-ms on SSD. The cost: every write must seek to where the key belongs in sorted order → random I/O, plus occasional page splits.

  • Bounded, predictable read & tail latency
  • Random writes (was brutal on HDD)
  • Page-level locks/latches on mutation
  • Postgres 8 KB, InnoDB 16 KB, Oracle, SQLite, lmdb

LSM-Tree — pay at read time + cleanup

write: append (sequential) · read: memtable + N SSTables

Writes hit RAM then flush sequentially as immutable SSTables → huge ingest throughput. A read may touch many files (memtable → newest → oldest); Bloom filters tame this by skipping files that can't hold the key. Counter-intuitively LSM write amplification is often higher (compaction rewrites data) — the win is writes are sequential & batched.

  • High write throughput, SSD-friendly
  • Reads/space amp paid back via compaction
  • Deletes = tombstones, not free
  • Cassandra, RocksDB, LevelDB, HBase, Bigtable, ScyllaDB

Things to know in the interview

  • "LSM is always faster" — wrong. LSM only optimizes the write path. Reads (without good Bloom filters, or on a missing key) can hit multiple SSTables; B-Tree reads are bounded to tree height and are often faster. A blanket "LSM faster" is a red flag.
  • The core insight: random vs sequential writes. B-Tree must write data where it belongs in sorted order → random I/O. LSM always appends forward → sequential I/O, sorting later via compaction. Game-changer on HDD, still wear/IOPS-friendly on SSD.
  • Write amplification is backwards from intuition. LSM's write amp is often higher than a B-Tree's (compaction rewrites data again and again). The advantage isn't lower amplification — it's that the writes are sequential and batched, giving high throughput and low latency.
  • Bloom filters are load-bearing. Explaining LSM reads without them is half a picture — they let reads skip SSTables that definitely don't hold a key (no false negatives, small false-positive rate).
  • Deletes aren't free. An LSM delete is itself a write — a tombstone. Data lingers until compaction; high-churn / TTL-heavy loads hit Cassandra's classic "tombstone hell".
  • Compaction is a hidden cost. It burns CPU + disk I/O and can spike foreground p99/p999 latency — a major production tuning axis, not free background magic.
  • WAL is in both. Both engines append to a WAL/commit-log first for durability — it does not distinguish the two designs.
  • Compaction strategies: Leveled (low read amp, high write amp) vs Size-tiered (low write amp, high read amp) — RocksDB offers both; pick by workload.

Key takeaway

Both store sorted data with a WAL, but they move the work to opposite ends: a B-Tree does the hard work at write time — find the sorted spot, update the page in-place (random I/O) — buying bounded, predictable reads. An LSM-Tree buffers in RAM and appends sequentially, deferring the work to reads + background compaction — buying high write throughput at the price of read/space amplification and compaction spikes. Choose by workload: write-heavy/high-ingest → LSM, read-heavy/low-latency OLTP → B-Tree, and always justify with the amplification trade-offs.
