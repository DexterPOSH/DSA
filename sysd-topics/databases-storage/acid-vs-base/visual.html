ACID vs BASE — Two Consistency Philosophies

The same write request, processed two ways. ACID coordinates first so every reader sees the truth instantly (but pays latency). BASE acknowledges first and lets replicas catch up eventually (but stays always-on). Watch one packet travel each path with Next / ←→.

Real-world analogy — bank transfer vs WhatsApp group

ACID = a bank transfer. Move ₹5000 from A to B and it's all-or-nothing: either A is debited AND B is credited, or neither. The moment it shows "done", every teller and every ATM sees the new balance. The bank may be a touch slow (locks, double-checks) — but the margin for error is zero.

BASE = a WhatsApp group. You hit send and instantly see "sent" — the system stays available even if members are offline. The message lands on each phone with a small lag (1s here, 5s there). For a few seconds different people see different "versions" (soft state), but eventually everyone has the same messages. Goal: never block, stay fast — over perfect instant agreement.

ACID — coordinate, then confirm

A·C·I·D = Atomicity, Consistency, Isolation, Durability
  • Atomicity — write-ahead log (WAL); crash → incomplete txn rolls back, no partial state.
  • Consistency — every constraint (FK, unique, balance>=0) holds, else abort.
  • Isolation — locks / MVCC make concurrent txns act serial.
  • Durability — fsync'd to disk/replicas; survives power loss.
  • Cost: ~1-10ms single-node, 10-100ms+ distributed 2PC; a down node can block commit. Usually CP.

BASE — confirm, then converge

BA·S·E = Basically Available, Soft state, Eventually consistent
  • Basically Available — always responds, even with stale data; reads/writes don't block if some replicas are down.
  • Soft state — state drifts on its own as replicas converge (read-repair, anti-entropy).
  • Eventually consistent — stop writing, and all replicas reach the same value (ms to ~1-2s, e.g. DynamoDB cross-region).
  • Quorum knobs N,W,R: W+R>N = strong-ish, W+R≤N = pure eventual + max throughput (100K+ QPS). Usually AP.

Things to know in the interview

  • "NoSQL = BASE, SQL = ACID" is wrong. These are consistency models, not data models. MongoDB does multi-document ACID; Spanner & CockroachDB are distributed yet fully ACID.
  • BASE doesn't mean data is lost. Writes are durably stored on multiple replicas — they're just not visible everywhere at the same instant. "Eventual" = temporarily different across replicas, not "lost".
  • Two meanings of "consistency". ACID's C = constraints / integrity rules. CAP's C = replicas agree (linearizability). Interviewers trap you here — they're different.
  • "ACID can't scale" is outdated. Spanner (TrueTime atomic clocks/GPS), CockroachDB, YugabyteDB give ACID + horizontal scale at coordination cost.
  • Eventual consistency needs conflict resolution. Concurrent writes collide → last-write-wins (can lose data), version vectors, or CRDTs. Saying "eventually consistent" isn't enough — explain how it converges.
  • Pattern match the prompt. "balance / payment / must not double-count" → ACID. "billions of users / global / feed / always-on / high write throughput" → BASE.

Key takeaway

ACID coordinates before it confirms — strong, instantly-consistent, all-or-nothing, but with locking/2PC latency (CP). BASE confirms before it coordinates — always-available and massively scalable, accepting brief stale reads while replicas converge (AP). It's not NoSQL-vs-SQL and not a binary — it's a tunable spectrum (per-query N/W/R, NewSQL); pick by whether correctness or availability/scale matters more for that data.
