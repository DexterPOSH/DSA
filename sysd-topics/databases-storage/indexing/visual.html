Indexing — B+ Tree Lookup vs Full-Table Scan

How a database serves WHERE email = 'a@b.com' on a 10-million-row table in a few milliseconds instead of seconds — by walking a shallow, sorted B+ tree instead of reading every row. Plus the price you pay: extra storage and slower writes.

Real-world analogy — the textbook index

You need to find "consistent hashing" in a 1000-page textbook. Option A: start at page 1 and read every page until you hit it — that's a full-table scan, O(n), worst case the whole book. Option B: flip to the index at the back, where important words are alphabetically sorted with page numbers beside them. You binary-search the sorted list, jump straight to page 814 — that's an index lookup. Two catches: the index itself eats extra paper (storage), and every time the publisher adds a paragraph they must re-update the index too (slower writes). Exactly the database tradeoff: fast reads, more space, costlier writes.

No index  full-table scan

read row 1, 2, 3 … n

No structure to exploit, so the DB checks every row. Across 10M rows that's a huge chunk of sequential disk reads — easily 100s of ms → seconds.

Cost grows linearly: O(n). Double the table, double the wait.

B+ tree index lookup

root → branch → leaf

High fan-out (100s of keys per node) keeps the tree shallow — even 10M rows fit in ~3–4 levels. Each level is one page-cache hit or disk seek, so the whole lookup is just a few ms.

Cost grows logarithmically: O(log n). 10× the table adds barely one level.

Things to know in the interview

  • Leaves are a linked list — leaf nodes are chained in sorted order, so WHERE age BETWEEN 25 AND 30 finds the start key then walks leaves sequentially. Sorted order gives ORDER BY and range scans for free.
  • Hash index — maps a key to a bucket: O(1) equality, even faster than B-tree. But hashing destroys order, so it's useless for BETWEEN, <, or ORDER BY.
  • Clustered vs secondary — a clustered index stores the rows themselves in the leaf (one per table). A secondary index's leaf holds key + a pointer, so it needs an extra lookup back to the clustered index — unless it's covering.
  • Covering / composite — if the index has every column the query needs (SELECT email … WHERE id=5 on index (id, email)), the DB never touches the table: an index-only scan, the fastest path. Composite indexes obey the leftmost-prefix rule — (a,b) helps WHERE a=? but not WHERE b=? alone.
  • Write cost — every INSERT/UPDATE/DELETE must maintain every affected index (possible node splits). A table with 5 indexes writes far slower than one with 1. Don't over-index write-heavy tables.
  • B-tree vs LSM-tree — B-tree is read-optimized (in-place updates). LSM-tree (Cassandra, RocksDB) is write-optimized: batches writes into a memtable + append-only SSTables, compacted in the background. Reads may scan several SSTables, mitigated by Bloom filters.

Key takeaway

An index is an auxiliary sorted structure (usually a B+ tree) that turns a WHERE/JOIN/ORDER BY lookup from O(n) full scan into an O(log n) walk of a shallow, sorted tree whose leaves are linked for range scans — paid for with extra storage and slower writes, so you index high-selectivity columns, not every column.
