Event-Driven Architecture — The Fan-Out Flow

How one service announces a fact (OrderPlaced) to a broker and many other services react independently and asynchronously — instead of the order-service calling each of them by hand. The producer never even knows who is listening.

Real-world analogy — the wedding baaja-band

Picture a shaadi. The band (the producer) starts playing — that's the event: "baraat nikli". They don't phone one specific person and say "come dance". They just bang the dhol, and whoever hears it reacts on their own: someone dances, someone grabs a camera, someone showers cash, the halwai goes to warm the gulab jamun. The band has no idea how many people will react. Add a new cameraman tomorrow — the band changes nothing; he just hears the dhol and starts. That's EDA: loose coupling — producer and consumers work without knowing each other, joined only by a shared event.

Synchronous  (direct calls)

order → inventory → email → analytics → fraud …

order-service must know and call every downstream and wait for each. Add a 6th reaction → change & redeploy order-service. One slow/down dependency blocks the whole request, and a traffic spike hammers all of them at once.

Coupling cost grows with each new reaction: O(N) calls wired by hand.

Event-driven (fan-out)

order → emit OrderPlaced → broker → N consumers

order-service emits one event and is done. Each consumer reacts on its own pace; broker buffers spikes (load leveling). Add a new consumer group → zero changes to the producer.

Producer effort stays O(1) no matter how many react.

Things to know in the interview

  • Event ≠ command. An event is a fact that already happened — OrderPlaced (past tense). A command is an instruction — PlaceOrder. The producer announces; it doesn't order anyone around. Past-tense naming reinforces decoupling.
  • Delivery is at-least-once by default. A network retry can deliver the same event twice, so consumers must be idempotent — dedup via eventId. Blindly doing balance += 100 on a duplicate = double charge.
  • Ordering is per-partition only, not global. Want all of one user's events in order? Use userId as the partition key so they land on the same partition. Watch for hot partitions on high-cardinality keys.
  • Offsets / acks track progress. Each consumer commits an offset (Kafka) or acks a message (queues). Crash → resume from last committed offset. No ack → redelivery (the root of at-least-once).
  • Scale = add partitions/consumers. 12 partitions ⇒ at most 12 active consumers in a group; extras sit idle. lag = latest offset − committed offset; growing lag = consumers can't keep up.
  • Failures are real. Retries with backoff, a DLQ for poison messages, and lag monitoring show maturity. The system is eventually consistent — "order placed" shows instantly, inventory may update seconds later.

Key takeaway

A producer publishes an immutable event to a durable, partitioned broker, and any number of consumer groups react independently and asynchronously — each getting its own copy and tracking its own offset. You get loose coupling, fan-out, and spike absorption; you pay with eventual consistency, so design consumers to be idempotent (at-least-once), key partitions for ordering, and add retries + a DLQ for failures.
