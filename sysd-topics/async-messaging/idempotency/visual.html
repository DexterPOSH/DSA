Idempotency — Safe Retries

In an at-least-once world the network can lose a response (not the request), so the client retries — and a naive server charges twice. An idempotency key + a server-side dedup store make the duplicate harmless: same key in, same saved response out, exactly one ₹500 charge.

Real-world analogy — the elevator call button

Press the up button and the lift is summoned. Nervously jab it 5 more times and… the lift still comes exactly once. The net effect of many presses equals one press — the button is idempotent.

Contrast a vending machine: every "Coke" press drops a can and charges ₹50. A laggy network + a double press = two cans, ₹100 — a duplicate action with a duplicate side-effect. That is the double-charge we must prevent. The idempotency key turns every critical write into a "lift button."

Naive  no idempotency

retry → charge() again

Response was lost, not the request — so the original charge already ran. The client's retry hits a server with no memory of K1 and runs charge() a second time.

Result: 2 × ₹500 = ₹1000 billed. The classic double-charge.

Idempotency key + dedup store

SET K1 ... NX → hit(done) → replay

The retry carries the same K1. The server sees status=done, skips charge(), and returns the saved {charge_id: ch_99}.

Result: 1 × ₹500 billed. At-least-once delivery + idempotent processing = effectively-once effect.

Things to know in the interview

  • Why duplicates are unavoidable — Kafka, SQS, gRPC retries and payment gateways all give at-least-once, never exactly-once: the network can't tell a lost response from a lost request. Absorbing the duplicate is the consumer's job.
  • The key must be fixed before the retry boundary — the client generates K1 once and reuses the same K1 on every retry. A fresh UUID per attempt = no dedup at all (the #1 practical bug).
  • Check-then-act is a race — naive SELECT-then-INSERT lets two concurrent duplicates both miss and both charge. Use an atomic INSERT ... ON CONFLICT DO NOTHING or Redis SET ... NX — whoever wins the insert processes; the loser replays.
  • Save the full response, not just a flag — store the first {charge_id: ch_99} so a duplicate gets exactly the same result, not a generic "already done."
  • Same key, different body → reject (409/422) — K1 with ₹500 then K1 with ₹900 is a client bug; silently replaying corrupts data. Stripe rejects it.
  • TTL the keys — keep keys 24h–7 days (Stripe honors ~24h) — long enough to cover realistic client + gateway retries without unbounded storage.
  • Naturally idempotent first — PUT /user/42, SET balance=500, DELETE /order/9 need no dedup store. Only relative / side-effectful ops (INSERT, balance += 100, "send email", charge) need the key.
  • Idempotent ≠ safe — GET is safe (no state change) and idempotent; DELETE is idempotent but not safe. And "PUT/DELETE idempotent, POST not" is HTTP semantics, not a guarantee — a sloppy PUT handler can break it.

Key takeaway

Make the client send a stable idempotency key with every mutating request, and have the server atomically record-and-act on that key (SET NX / ON CONFLICT) while saving the first response. Then a lost-response retry replays the saved result instead of re-running the side-effect — turning unavoidable at-least-once duplicates into an effectively-once effect, with no double-charge.
