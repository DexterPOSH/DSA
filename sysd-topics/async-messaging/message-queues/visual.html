Message Queues — Producer, Broker & Competing Consumers

How a queue lets a producer hand off work and walk away, while a broker buffers the burst and competing consumers drain it — each message going to exactly one worker, with ack, visibility timeout and a dead letter queue keeping delivery safe.

Real-world analogy — the kitchen order rail

In a busy restaurant the waiter (producer) doesn't stand next to the chef waiting for the dish — they clip the order ticket onto the order rail (the queue) and immediately go serve the next table. A free chef (consumer) grabs the next ticket off the rail and cooks it. If lunch-rush orders arrive faster than chefs can cook, tickets just pile up on the rail (buffering the burst) — nothing is lost. One ticket is taken by exactly one chef so no dish is cooked twice, and if a chef calls in sick another chef simply works the same rail. Producer and consumer are decoupled in both time and speed.

Queue — point to point

1 message → 1 consumer

Competing consumers share the work. Add more consumers and throughput goes up — each message is handled exactly once across the pool. Use when a task must be done once: send the email, charge the card, transcode the video.

Tools: AWS SQS, RabbitMQ, ActiveMQ, Redis Streams.

Pub/Sub — fan-out

1 message → ALL subscribers

A topic broadcasts the event to every independent subscriber. One order_placed event reaches inventory + shipping + analytics + email at once. Use when many systems must each react to the same event.

Tools: Kafka, AWS SNS, Google Pub/Sub.

Things to know in the interview

  • At-least-once, not exactly-once. An ack can be lost or a consumer can crash mid-process, so a message may be redelivered. Make consumers idempotent (dedup on order_id) or you get double charges / double emails — the single most common trap.
  • Visibility timeout. On receive, the broker hides the message (SQS default 30s). If no ack arrives in time it becomes visible again for another consumer. Set it above worst-case processing time, or you redeliver work that's still running.
  • Ordering isn't free. Standard SQS / multi-partition Kafka give no global order. Need strict FIFO? Use a FIFO queue (~3000 msg/s) or a single Kafka partition — and accept the lower throughput.
  • DLQ stops poison messages. A message that always fails would block the queue (especially FIFO). After max-retries (e.g. 5) route it to a dead letter queue to inspect/replay later.
  • The queue is not an infinite buffer. If producers are sustainably faster than consumers the backlog grows forever — that's a bug. Scale consumers or apply backpressure; a queue only absorbs temporary bursts.
  • Queue vs log. SQS/RabbitMQ delete on ack. Kafka is an append-only log — messages stay for a retention window, consumers track an offset, and multiple groups can replay the same data independently.

Key takeaway

A message queue puts a durable broker between producer and consumer so they're decoupled in time and speed: the producer fires-and-forgets, the queue buffers the burst, and competing consumers each pull exactly one message and ack it. Because delivery is at-least-once, keep consumers idempotent, tune the visibility timeout, and route poison messages to a DLQ.
