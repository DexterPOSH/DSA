Pub/Sub — One Publish, Many Deliveries

How a publisher drops a message onto a named topic and a broker fans it out to many independent subscribers — sender and receivers fully decoupled, never knowing each other exist.

Real-world analogy — a YouTube channel

A creator (the publisher) uploads a video to their channel (the topic). They have no idea who will watch, how many, or when. Everyone subscribed to that channel (the subscribers) gets the same video and watches on their own time. One viewer skipping it doesn't affect anyone else, and a brand-new subscriber tomorrow still gets future uploads. YouTube (the broker) sits in the middle so creator and viewers never deal with each other directly: one publish, many independent deliveries (fan-out), both sides scaling on their own.

Point-to-point queue

1 message → exactly 1 consumer

A plain queue is for work distribution: each message is handled once, load-balanced across consumers. Great for "process this job", but only one worker ever sees it.

In Kafka terms: a single consumer group behaves queue-like.

Pub/Sub (fan-out)

1 message → every subscribing group

The same message is broadcast: each interested group gets its own copy. Add a new subscriber and it just starts receiving — publisher unchanged.

In Kafka terms: multiple consumer groups = pub/sub-like.

Things to know in the interview

  • Decoupling — the publisher knows only the topic name. It publishes and moves on; it never blocks on subscriber availability or speed. A modern broker (Kafka) sustains millions of messages/sec with single-digit-ms publish p99.
  • Partitions cap parallelism — a consumer group's parallelism is bounded by partition count. 12 partitions + 4 consumers = 3 each; a 13th consumer in that group sits idle. To scale further, add partitions.
  • Ordering is per-partition only — no global order across a topic. Need related events in order (e.g. one user's events)? Use a partition key like userId so they land on the same partition. Strong ordering trades off parallelism.
  • Delivery guarantees — at-least-once is the common default: if a consumer crashes before ack/offset-commit, the broker redelivers → duplicates possible → make consumers idempotent. True "exactly-once" across a network is a myth (at-least-once + dedup).
  • Log-based vs push — Kafka retains messages (e.g. 7 days), so a new subscriber can replay history from an offset. Redis Pub/Sub is fire-and-forget: offline subscriber → message lost. "Pub/Sub" does NOT imply durability.
  • Watch consumer lag — if consumers fall behind producers and retention is short, messages drop out of the window. Monitor the consumer-lag metric.

Key takeaway

A publisher drops a message on a named topic and a broker fans that one message out to every subscribing group, each consuming independently at its own pace via offsets. Sender and receivers are fully decoupled — but remember the fine print: ordering is only per-partition, delivery is usually at-least-once (so go idempotent), and durability/replay exists only with a log-based broker like Kafka.
