Kafka Basics — The Distributed Commit Log

How a topic is split into ordered partitions, how a producer keys records to a partition, how every record gets a permanent offset, how partitions are replicated for durability, and how a consumer group spreads partitions across consumers — so max parallelism = partition count.

Real-world analogy — the national newspaper

A topic is a newspaper like "Times of India". One press can't print for the whole country, so it has several partitions — regional editions (Mumbai, Delhi, Chennai). Each edition is an append-only log: today's issue #5001, tomorrow's #5002 — new issues only get added at the end, and each issue's number never changes (that's the offset). Producers are reporters: a story tagged city=Mumbai (the key) always lands in the Mumbai edition, keeping that city's news in order. Consumers are subscribers — Kafka never deletes a read paper; the backlog sits on the shelf for the retention window, and each subscriber keeps their own bookmark. Many independent subscribers (analytics, billing, search) read the same edition without disturbing each other: written once, read many times independently.

Traditional queue (RabbitMQ / SQS)

ack a message → it's gone

A message is delivered, acked, and deleted. One logical consumer drains the queue; "replay last 3 days" is impossible — the data no longer exists. Great for task-queue routing, not for re-reading history.

Kafka — a distributed log

read advances an offset; data stays

Records are not deleted on read — they live for the retention window. Each consumer group keeps its own offset, so N groups read the same data independently and replay is just "seek the offset back". Pub/sub fan-out for free.

Things to know in the interview

  • Ordering is per-partition, not global. The single most common mistake. Need global order → use one partition (parallelism = 1). In practice, partition by key so related events (same user/order) stay ordered, which covers 99% of cases.
  • Max parallelism = partition count. One partition goes to exactly one consumer within a group. 4 partitions, 6 consumers → 2 sit idle. Interviewers love "what's the scaling limit?".
  • Multiple groups, same data. Group-A (fraud) and Group-B (warehouse ETL) each read every event with their own offset — that's the pub/sub fan-out.
  • acks ≠ replication factor. Replication factor (topic-level) is how many replicas exist; acks (producer-level) is how many must confirm a write. acks=all means "all ISR", and min.insync.replicas=2 sets the durability floor.
  • Default delivery is at-least-once. Exactly-once needs idempotent producer + transactions and only across a Kafka-to-Kafka boundary; most systems prefer at-least-once + idempotent consumers (dedup on a key).
  • Retention ≠ consumption. Records expire on the retention window whether consumed or not. A slow consumer that falls behind retention loses data (offset out of range) — that's consumer lag, monitor it.
  • Partitions only grow. You can add partitions but not remove them, and adding them changes where future keyed records land (per-key ordering can break). Plan capacity upfront.

Key takeaway

Kafka is a distributed, append-only commit log, not a queue. A topic is split into partitions (the unit of ordering and parallelism); a producer routes a record by hash(key) % numPartitions and it gets a permanent offset; partitions are replicated to ISR for durability (acks=all); and a consumer group assigns one partition per consumer — so reads scale up to the partition count, multiple groups read the same data independently, and nothing is deleted on read.
