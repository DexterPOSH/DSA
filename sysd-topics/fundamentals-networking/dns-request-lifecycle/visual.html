DNS & HTTP Request Lifecycle

What actually happens when you type https://www.example.com and press Enter — from cache checks, through the recursive resolver's root → TLD → authoritative walk, to the TCP + TLS handshakes and the final HTTP response.

Real-world analogy — looking up a phone number

You need someone's extension. First you flip through your own contacts (the browser cache) — maybe you already wrote it down. If not, you check the sticky note on your monitor (the OS cache / /etc/hosts). Still nothing? You phone a friend who is great with directories (the recursive resolver) and say "just get me the number." That friend does the legwork: they ask the master directory operator (root) "who handles .com?", then ask that city directory (the TLD) "who handles example.com?", and finally call that company's front desk (the authoritative nameserver) for the exact extension (the IP). Your friend writes the number on their own pad for next time (caching, until the TTL expires) and reads it back to you. Only then do you actually dial.

Cold lookup (cache miss)

~tens–hundreds of ms

Nothing cached, so the resolver makes several sequential round trips before you even open a socket:

  • resolver → root (who owns .com?)
  • resolver → TLD (who owns example.com?)
  • resolver → authoritative (the actual A record)
  • then TCP (1 RTT) + TLS (1–2 RTT) + the HTTP request

Warm lookup (cache hit)

~0 ms for DNS

If the browser, OS, or resolver still holds a non-expired record (within its TTL), the whole hierarchy walk is skipped — the IP is returned instantly. The expensive part of "loading a page" is rarely DNS once it's cached.

  • TTL = how many seconds an answer may be cached
  • low TTL → fast failover, more lookups; high TTL → fewer lookups, slower changes

DNS record types you should name

  • A — hostname → IPv4 address (e.g. 93.184.216.34).
  • AAAA — hostname → IPv6 address.
  • CNAME — alias one name to another (www → example.com); resolver re-resolves the target.
  • NS — delegates a zone to its nameservers (this is what root and TLD hand back).
  • MX — mail exchanger for the domain (used by email, not web).
  • TXT — arbitrary text (SPF, domain verification, etc.).

Distribution & resilience tricks

  • CDN / GeoDNS — the authoritative server can return a different IP based on where the resolver is, sending you to the nearest edge PoP.
  • Anycast — one IP advertised from many locations; BGP routes you to the closest instance. Used by big resolvers (8.8.8.8, 1.1.1.1) and root servers.
  • Caching layers — browser → OS → resolver each cache by TTL, so most lookups never reach the authoritative server.
  • Load balancer — behind one IP, an LB fans requests across many origin servers.
  • Connection reuse — keep-alive / HTTP-2 multiplexing avoids redoing TCP + TLS for every asset.

Key takeaway

Typing a URL triggers two distinct phases: first name → IP resolution, where caches are checked (browser → OS) and, on a miss, a recursive resolver walks the hierarchy root → TLD → authoritative and caches the answer by TTL; then IP → page, where the browser opens a TCP connection, negotiates TLS, sends an HTTP GET (often to a nearby CDN edge), and renders the response. Cached DNS is ~0 ms; a cold lookup is several sequential round trips before a single byte of the page moves.
