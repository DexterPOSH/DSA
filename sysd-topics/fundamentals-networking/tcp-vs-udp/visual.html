TCP vs UDP — Reliability vs Speed

Watch the exact mechanics that make TCP a connection-oriented, reliable, ordered transport — handshake, ACKs, retransmission, head-of-line blocking — and why UDP's fire-and-forget minimalism is faster but offers zero guarantees.

Real-world analogy — registered courier vs postcard drop

TCP is a registered courier (Blue Dart with proof-of-delivery). First the courier calls to confirm you're home (handshake), then takes your signature for every parcel (acknowledgement). If a parcel is lost, they re-send it (retransmission). Order 3 boxes and they guarantee box 1, then 2, then 3 in order (in-order delivery). Reliable — but every confirmation step adds overhead, so it's slower.

UDP is a plain postcard drop. The postman just slips it through your letterbox and leaves — no signature, no "did it arrive?", no guarantee 5 postcards land in order. Lost means lost; no resend. But with zero tracking overhead it's fast and lightweight — like live cricket commentary, where re-sending a one-second-old audio packet is pointless because the match has already moved on.

TCP — reliability machinery

SYN → SYN-ACK → ACK, then seq# + ACK + retransmit

Pays a full round-trip handshake before the first byte (~50 ms same-region; ~100-150 ms cross-continent), keeps per-connection state, and a single lost segment can block the whole stream (head-of-line blocking).

Header ≥ 20 bytes. Use when correctness > speed: HTTP/1.1 & HTTP/2, gRPC, DB wire protocols, SMTP, SSH, file transfer.

UDP — minimalism

sendto() → done. no setup, no ACK, no order

Zero setup RTT, stateless (one server serves millions cheaply — why DNS uses it), datagram boundaries preserved (one recvfrom() = one datagram), and each packet is independent so a drop doesn't block others.

Things to know in the interview

  • Handshake cost is real — TCP 3-way handshake + TLS handshake on a high-RTT link can add 2-3 RTTs ≈ 300+ ms before the first byte. That's exactly why QUIC's 0-RTT / 1-RTT setup is a big deal.
  • Ordering is byte-stream, not message — TCP delivers a byte stream in order but does not preserve message boundaries; two send()s can arrive in one recv(). Need boundaries? Add your own framing (length-prefix/delimiter). UDP does preserve datagram boundaries.
  • Head-of-line blocking — one lost TCP segment makes later segments wait in the buffer until the retransmit arrives. HTTP/2 multiplexed streams over one TCP connection still hit this, which is why HTTP/3 (QUIC over UDP) does per-stream reliability.
  • UDP still has a checksum — it detects corruption (corrupt datagram is dropped) but never recovers/retransmits. Optional on IPv4, mandatory on IPv6.
  • Reliability ≠ security — TCP gives reliable delivery, not encryption. Encryption comes from TLS (over TCP) or QUIC's built-in TLS 1.3.
  • QUIC is the hybrid — built on UDP, it adds reliability, ordering, congestion control and TLS 1.3 per-stream, dodging TCP's HOL blocking. It's the transport for HTTP/3 (Google, Cloudflare, YouTube).

Key takeaway

TCP trades latency for guarantees: a handshake, ACKs, retransmission and in-order delivery give reliability but cost round-trips, per-connection state, and head-of-line blocking. UDP throws all of that away for a fire-and-forget 8-byte datagram — faster and stateless, but delivery, order and dedup become your application's job. Pick TCP when a lost byte breaks correctness; pick UDP when freshness and low latency beat perfect delivery.
