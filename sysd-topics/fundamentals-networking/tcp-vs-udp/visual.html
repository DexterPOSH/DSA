🌐 TCP vs UDP

Reliability vs Speed — the fundamental network protocol tradeoff

📦 Real-world analogy: TCP is like a certified letter — you get confirmation of delivery, packages arrive in order, and lost ones are re-sent. UDP is like shouting across a crowded room — instant, but you don't know who heard, and you never repeat yourself. Choose TCP when every word matters; choose UDP when freshness beats completeness.

🚧 TCP Retransmission & Head-of-Line Blocking

TCP guarantees in-order delivery. A single lost packet freezes every packet that follows it in the kernel buffer — even if the app could use them right now.

Key insight: HOL block duration = 1–3× RTO (200ms–3s on lossy links). HTTP/2 multiplexing over a single TCP still suffers this — one lost packet stalls ALL streams at the OS level. QUIC solves it with per-stream userspace retransmission.

🔥 UDP — Fire & Forget

UDP sends packets with zero setup overhead, no ACKs, no retransmission. Lost packets stay lost. Later packets never wait.

✅ UDP Strengths
• Zero connection overhead (0 RTT)
• No HOL blocking ever
• Works with multicast / broadcast
• App controls retry strategy
• 8-byte header vs 20+ bytes
⚠️ UDP Tradeoffs
• No delivery guarantee
• Out-of-order packets possible
• No built-in congestion control
• App must handle errors / gaps
• Duplicate packets possible
Key insight: A dropped video frame causes a 30ms visual glitch nobody notices. A dropped byte in a DB query silently corrupts data. Protocol choice = data criticality × latency tolerance.

📐 TCP vs UDP Header Anatomy

Every packet carries a header. TCP's 20-byte header buys reliability; UDP's 8-byte header buys speed.

🔵 TCP Header

20 bytes minimum (up to 60 with options)
Source Port16 bitsOrigin process
Dest Port16 bitsTarget process
Sequence Number32 bitsIn-order delivery
Ack Number32 bitsNext byte expected
Data Offset4 bitsHeader length
Control Flags9 bitsSYN/ACK/FIN/RST/PSH…
Window Size16 bitsReceive buffer / flow control
Checksum16 bitsError detection
Urgent Pointer16 bitsPriority data offset
Options0–320 bitsMSS, SACK, timestamps…
Fields in blue enable reliability, ordering & flow control — absent entirely in UDP.

🟠 UDP Header

8 bytes — always, no exceptions
Source Port16 bitsOrigin process
Dest Port16 bitsTarget process
Length16 bitsHeader + data in bytes
Checksum16 bitsError detection (optional IPv4)
That's it. 4 fields, 8 bytes. No seq, no ACK, no window, no flags. Pure simplicity.

📦 Header overhead on a 1-byte payload
TCP header: 20 bytes
→ 95.2% overhead
UDP: 8 bytes
→ 88.9% overhead — 2.5× less than TCP
Both are further wrapped by IP (20B) + Ethernet (14B+4B CRC) per frame — unavoidable regardless of transport.
Key insight: TCP's 12 extra bytes encode an entire reliability mechanism — seq/ack numbers track ordering, window size prevents buffer overflow, flags drive the state machine. Remarkably compact for what it delivers.

🗺️ When to Use TCP vs UDP

The decision hinges on one question: is a stale/missing message better or worse than a delayed one?
