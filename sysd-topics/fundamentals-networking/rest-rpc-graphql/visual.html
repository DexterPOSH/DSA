REST vs RPC vs GraphQL

The same job — render a user dashboard showing the user, their orders, and recommendations — done three ways. Watch the request packets travel and count the round-trips, payload shape, and caching cost for each style. ~50–150 ms of mobile latency rides on every round-trip.

Real-world analogy — three ways to order at a restaurant

REST is a printed menu of fixed dishes (resources). You talk in nouns + standard verbs — GET = "show me", POST = "make new", DELETE = "remove". Any waiter understands you. But want a dish plus its chef plus its calories? You may have to flag the waiter down three separate times.

RPC is calling a private cook directly: makeButterChicken(spicy=true). You focus on the verb, share a tight shorthand, and it's blazing fast — but a stranger who doesn't know the shorthand is lost (tight coupling).

GraphQL is handing a waiter one slip that says exactly which fields you want — name, price, chef — no more, no less. One round-trip brings precisely that, but the waiter needs a smart brain (query planner + resolvers) to fulfil any custom slip.

Things to know in the interview

  • "REST means JSON" — wrong. REST is an architectural style (resources, stateless, uniform interface, HTTP verbs), not a format. JSON is just the common convention; true REST also includes hypermedia (HATEOAS).
  • REST over- AND under-fetches. A thin endpoint under-fetches (extra calls); a fat endpoint over-fetches (extra bytes). GraphQL solves both by letting the client pick exact fields.
  • "GraphQL is always faster" — wrong. It saves round-trips and trims payload, but naive resolvers hit the N+1 problem — one query can fire hundreds of DB calls. DataLoader batching is almost mandatory.
  • gRPC runs on HTTP/2 (not HTTP/3 by default) with Protobuf binary — 30–60% smaller, single-digit ms internal latency. Browsers need a gRPC-Web proxy.
  • Caching: REST GETs cache free on CDN/ETag. GraphQL/gRPC are POST-style → need persisted queries, field-level caching, or query hashing.
  • Versioning: REST uses URL versions (/v1, /v2); GraphQL evolves fields with @deprecated; gRPC uses Protobuf field numbers for backward compatibility.
  • Not mutually exclusive: common production shape is gRPC between internal services + a BFF/gateway exposing REST or GraphQL to the outside.

Key takeaway

All three move data between client and server, but optimize different costs: REST trades round-trips for dead-simple, cacheable, universally-understood resource URLs; gRPC trades browser-friendliness for the fastest, most compact, strongly-typed internal calls; GraphQL trades caching simplicity for one round-trip that returns exactly the fields each screen needs. Pick by who the client is and what you're optimizing — and remember they happily coexist layer-by-layer in one system.
