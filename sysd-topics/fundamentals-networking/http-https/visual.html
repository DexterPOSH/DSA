HTTP & HTTPS — The Request/Response Cycle & TLS Handshake

How a client and server talk over the wire: a stateless request → response exchange, and how HTTPS wraps it in a TLS handshake so the conversation is private, tamper-evident, and provably with the real server.

Real-world analogy — postcard vs. sealed envelope

Plain HTTP is like mailing a postcard. You write your message (the request), put an address (the URL), and the postman — the network of routers, ISPs and proxies — carries it. Trouble is, every postman and sorting office along the way can read it, and even change it, because everything is in the open as plain text.

HTTPS seals that postcard inside a tamper-evident envelope whose key only you and the real receiver hold. A nosey postman can't read it (confidentiality), and any meddling is instantly detectable (integrity). Crucially, before you seal it you check the receiver's ID card — the certificate — so no imposter can stand in the middle pretending to be your bank (authentication). That ID check is exactly what the TLS handshake does.

HTTP/1.1

1 request → wait → next request

One request at a time per connection → head-of-line blocking. Browsers work around it by opening 6 parallel TCP connections per host.

HTTP/2

N streams over 1 TCP connection

Multiplexed streams + header compression (HPACK) on a single connection. Kills app-level HoL — but one lost TCP packet still stalls all streams (TCP-level HoL).

HTTP/3 (QUIC)

independent streams over UDP

Runs on QUIC over UDP with TLS 1.3 built in and 0/1-RTT setup. A lost packet hits only its own stream — best on lossy/mobile networks. Used by Google, Cloudflare, YouTube.

Things to know in the interview

  • HTTPS is three things, not one. Confidentiality (encryption), integrity (tampering is detectable), and authentication (the server is who it claims, via certificate + CA trust chain). The authentication part is what stops MITM attacks — people forget it.
  • The certificate is only used in the handshake. Its asymmetric key authenticates the server and helps exchange a symmetric session key; actual data is encrypted with that fast symmetric key (AES-GCM). Asymmetric crypto on every byte would be far too slow.
  • "HTTPS is slow" is mostly myth. TLS 1.3 adds just 1 RTT (resumed connections can be 0-RTT), and AES is hardware-accelerated. With keep-alive/connection reuse the per-request overhead is ~zero.
  • Stateless by design. The server links no two requests by default; login/cart state rides along in cookies, tokens, or session IDs. This is what lets servers scale horizontally — any server can take any request.
  • Idempotency drives retries. GET, PUT, DELETE, HEAD are idempotent (retry-safe); POST is not — two POSTs can create two resources.
  • 301 vs 302. 301 permanent is cached by clients (future requests skip straight to the new URL); 302/307 temporary is not. Use 301 by mistake and clients cache the stale redirect for a long time.
  • Where TLS terminates. Often at a load balancer / reverse proxy (NGINX, Envoy, ALB), then plain HTTP internally — simple and CPU-efficient. Zero-trust meshes (Istio) use end-to-end mTLS instead, at higher operational cost.

Key takeaway

HTTP is a stateless request/response protocol — a method + path + headers go out, a status code + body come back, over TCP. HTTPS = HTTP + a TLS handshake: verify the server's certificate, agree on a symmetric session key, then encrypt everything. TLS 1.3 costs only 1 extra RTT (0-RTT when resumed), so on today's hardware HTTPS everywhere is essentially free — and non-negotiable.
