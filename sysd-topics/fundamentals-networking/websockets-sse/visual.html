WebSockets & SSE — Two Ways to Keep a Connection Open

Both let a server push data without the client re-asking. Watch the message flow on a single persistent connection: WebSocket is a full-duplex phone call (ws://), SSE is a one-way radio broadcast over plain HTTP (text/event-stream).

Real-world analogy — phone call vs radio channel

WebSocket = a phone call. Once the call connects (the handshake), both people can speak freely, anytime, without redialing. The line stays open until someone hangs up. Bidirectional and low-latency — the moment you speak, the other side hears you.

SSE = a news radio channel. You tune in once (one HTTP request) and the station keeps broadcasting updates to you. You can only listen — to talk back you must make a separate phone call (a normal HTTP request). And if the signal drops, your radio re-tunes itself automatically — that's SSE's built-in Last-Event-ID reconnect.

WebSocket — the phone call

GET … Upgrade: websocket → 101 → ws:// frames
  • Full-duplex: client and server both send anytime on one connection.
  • Tiny frame headers (2–14 B) vs hundreds of bytes per HTTP request → great for high-frequency messaging.
  • Latency ~ one round trip (same-region single-digit to low double-digit ms).
  • ping/pong frames (~every 30s) keep the line alive through proxies.
  • Reconnect, backoff & replay are your job. Prefer wss://.
  • Use for chat, multiplayer, collaborative cursors, live order books.

SSE — the radio broadcast

GET Accept: text/event-stream → data: …\n\n
  • Unidirectional: server → client only; uplink needs a separate HTTP request.
  • Plain HTTP — works with existing LBs, proxies, auth, HTTP/2. No upgrade.
  • Browser EventSource auto-reconnects & sends Last-Event-ID to resume.
  • UTF-8 text only — binary must be base64 encoded.
  • HTTP/1.1 gotcha: max 6 connections/domain; HTTP/2 multiplexing fixes it.
  • Use for notifications, live feeds, progress logs, LLM token streaming.

Things to know in the interview

  • "WebSocket is always better" — wrong. Stateful persistent connections cost the server memory + a file descriptor each, and add infra complexity. For pure server-push, SSE is simpler, cheaper and HTTP-native.
  • SSE is not bidirectional. The client can't push over the stream — uplink needs a separate HTTP request. Need both directions → WebSocket.
  • Stateful scaling. Both pin a client to one server instance, so you need sticky sessions + a Pub/Sub layer (e.g. Redis Pub/Sub) to broadcast a message to clients connected to other instances.
  • Proxy/LB timeouts kill idle long-lived connections — hence WebSocket ping/pong and SSE heartbeat comment lines : keepalive\n\n.
  • HTTP/1.1 6-connection limit bites SSE across multiple tabs; fix with HTTP/2 or a shared connection (SharedWorker). WebSocket isn't under this cap.
  • Long-polling is the older fallback when neither is available: hold the request open, return on data, client immediately re-requests — higher latency & overhead, most compatible.

Key takeaway

Both keep one connection open so the server can push. Ask "does the client need to actively send to the server?" — if yes, WebSocket (full-duplex frames after an Upgrade handshake). If it's pure server → client and you want HTTP-native simplicity with free auto-reconnect, SSE (text/event-stream). Don't reach for WebSocket's stateful complexity when a radio broadcast will do.
