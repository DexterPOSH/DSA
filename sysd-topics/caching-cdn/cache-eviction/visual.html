Cache Eviction — LRU & LFU

A cache is a bounded box of C entries living in RAM (~1 ms lookup) in front of a slow origin (50–200 ms). When it fills up, the eviction policy decides which entry to throw out to make room: LRU drops the least-recently-used, LFU drops the least-frequently-used.

Real-world analogy — the study table with 3 book slots

Your desk holds only a few books (the cache capacity). The full library (the origin / DB) is in a slow basement. When the desk is full and you need a new book, you must send one back downstairs.

LRU librarian: "Return whichever book you haven't touched for the longest." This morning's book stays; the one untouched for two weeks goes first. Bet: recently used → likely used again soon.

LFU librarian: keeps a tally mark on each book per read and returns the one with the fewest total reads. A dictionary you've opened 50 times survives even if untouched today; a novel read once leaves first.

LRU — Least Recently Used

hash map + doubly linked list → O(1)

Hit: splice node out, move to head. Miss: insert at head; if size > C, drop the tail + delete its map key. Every op is just 2–3 pointer updates — microseconds on a 1000-entry cache.

Great at temporal locality. Weakness: a one-time scan/flood (full table read) pushes all hot entries to the tail and evicts them — cache pollution.

LFU — Least Frequently Used

freq buckets + minFreq pointer → O(1)

Each entry carries a counter; access does ++ and moves it to the next freq bucket. Evict the tail of the minFreq bucket. Naive "scan for lowest count" is O(N) — buckets make it O(1).

Captures long-term popularity and resists scans (one-time reads stay at freq 1). Weakness: stale once-popular entries get stuck → need aging / decay (W-TinyLFU).

Things to know in the interview

  • "LRU is O(1) — how?" Not a plain hash map. You need hash map + doubly linked list together: map gives key → node in O(1), the doubly linked list makes detach + move-to-head + tail-eviction O(1). A singly linked list would make detach O(N).
  • LRU vs LFU. LRU = recency (when last used), LFU = frequency (how often). On a scan workload LRU pollutes, LFU holds; on shifting popularity LFU sticks on stale entries, LRU adapts.
  • "Naive LFU is O(1)" is wrong. Finding the lowest count by scanning is O(N). True O(1) LFU = per-frequency buckets + a minFreq pointer; new entries enter at freq 1, minFreq = 1.
  • Redis isn't exact. allkeys-lru / allkeys-lfu use sampling (maxmemory-samples, default 5) — evict the oldest of N random keys, not the true global oldest. Linux page cache approximates LRU with CLOCK / second-chance (a reference bit).
  • Admission control. Modern caches (Caffeine = W-TinyLFU) admit a new item only if a frequency sketch says it's more valuable than the victim — beating plain LRU's hit ratio.
  • Don't confuse with stampede. Eviction = making room. Stampede / thundering herd = one hot key expiring and every request hitting origin at once → fixed by request coalescing, not by policy choice.

Key takeaway

A bounded cache must pick a victim when full. LRU bets on recency and gets O(1) from a hash map + doubly linked list (move-to-head on hit, drop the tail on overflow); LFU bets on frequency and gets O(1) from freq buckets + a minFreq pointer. Lifting the hit ratio even from 90%→95% halves origin load — so the policy choice directly drives tail latency and backend cost.
