Cache Strategies — How Reads & Writes Flow

Patterns that decide how the application, the cache (Redis/Memcached, ~1 ms) and the database (5-20 ms) route and synchronize data. Each one trades speed against consistency against data-loss risk.

Real-world analogy — the restaurant kitchen

The main store-room (database) is far at the back and slow. At the counter the cook keeps a small shelf (cache) with the most-used ingredients. Cache-aside: the cook checks the shelf first — found it (hit), else walks to the store-room, grabs it, drops a copy on the shelf (miss). Write-through: new stock goes onto the shelf and the store-room together, so they always match. Write-back: the cook updates only the shelf and syncs the store-room later in a batch — fast, but if the shelf tips over, pending updates are lost.

Write-through

write → cache + DB (synchronous)

Every write lands in both cache and DB before success returns. Cache is always consistent with the DB, with no data-loss on a cache crash.

Cost: write latency = cache + DB (the sum). Writes are slower. Good for config / session data where read-after-write must hold.

Write-back (write-behind)

write → cache (ack now), DB flushed async

Write hits only the cache and acks in ~1 ms. Dirty entries are batched and flushed to the DB later — best throughput, absorbs write bursts.

Cost: a cache crash loses un-flushed writes (data-loss risk). Great for counters / metrics; never for financial data.

Things to know in the interview

  • "Cache is always consistent with the DB" — false. Only write-through (sync) gives tight consistency. Cache-aside and write-back always have a staleness window. State the consistency model explicitly.
  • Invalidate, don't update, on write. On a write, delete the cache key rather than overwriting it — concurrent writes can overwrite the cache with a stale value (race). Delete forces the next read to reload fresh.
  • Cache stampede / thundering herd — a popular key expires and thousands of concurrent misses slam the DB at once. Mitigate with request coalescing / single-flight, early/probabilistic expiration, or a short-TTL lock.
  • Negative caching — cache "not found" with a short TTL too, or missing keys hammer the DB on every request.
  • Write-around — writes go straight to the DB, bypassing the cache, so write-once-read-never data doesn't pollute it. Tradeoff: the first read of fresh data always misses.
  • Read-through — like cache-aside, but the cache library loads from the DB on a miss, not your app code. Cleaner code, less control. e.g. Netflix EVCache; cache-aside powers Facebook's Memcached + MySQL.

Key takeaway

There is no "best" cache strategy — only the right trade. Cache-aside is the read-heavy default (lazy-load on miss, invalidate on write). For writes you choose your poison: write-through buys consistency at the cost of slower writes, write-back buys blazing throughput at the risk of lost data, and write-around keeps rarely-read data out of the cache. Always pair them with a TTL + LRU and a plan for the stampede.
