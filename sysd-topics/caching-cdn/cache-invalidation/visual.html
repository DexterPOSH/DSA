Cache Invalidation & Staleness

How a fast cache layer is kept in sync with the source-of-truth database — and the staleness window during which readers might still see old data. Watch a request packet travel between Reader, Cache, Writer, and DB across four scenarios.

Real-world analogy — the canteen notice board

A notice board (the cache) shows the canteen menu so people read it fast — no trip to the kitchen. The real menu lives in the kitchen (the source-of-truth DB). The catch: when the kitchen changes the menu, anyone reading the un-updated board sees a stale menu. Three ways to re-sync: write "valid till 10am" on the board so readers refresh it themselves (TTL), tear the old slip off the moment the kitchen changes (invalidate on write), or have the kitchen pin a fresh slip immediately (write-through). Cache invalidation is the whole game of keeping that board↔kitchen gap small.

Delete-then-update DB  (wrong order)

DEL cache  →  (reader repopulates)  →  UPDATE db

A reader slips in between the delete and the DB write, reads the old DB value, and re-caches it. The cache is now stale until the next write — possibly forever.

Result: a silent, long-lived mismatch between cache and source.

Update DB-then-delete cache  (industry standard)

UPDATE db  →  DEL cache

The next read misses and pulls the fresh value from the DB. The bad window shrinks to ~ms, and a short TTL backstops any missed delete.

Result: self-healing — even a lost invalidation event expires on its own.

Things to know in the interview

  • TTL is the staleness window. TTL = 300s means a reader can see data up to 5 minutes old after the source changed. Small TTL → fresher data but more misses & backend QPS; big TTL → less load but more staleness.
  • Delete vs update on write. DEL (invalidate) is safer — next read pulls truth — at the cost of one extra miss. SET (overwrite) is faster but concurrent writers can leave a stale value behind.
  • Cache-aside ordering. Always update DB, then delete cache. For edge races add a delayed double-delete or rely on a short TTL safety net.
  • Thundering herd / stampede. One hot key expiring sends thousands of misses at the DB at once. Fix with single-flight / request coalescing, early probabilistic refresh, or stale-while-revalidate.
  • Invalidation is best-effort. Network drops & consumer lag can lose an event — keep TTL as a backstop, never trust invalidation alone.
  • CDN: purge vs versioning. Purge tells edge POPs to drop a URL (propagates seconds→minutes, eventually consistent). Versioned URLs (app.abc123.js) are instant & globally consistent since a new name is a brand-new object.
  • Used by — Facebook TAO + Memcache (cache-aside), Netflix EVCache, Stack Overflow (Redis + targeted invalidation), CDNs Cloudflare / Akamai / Fastly (Fastly claims ~150ms instant-purge).

Key takeaway

A cache is only correct if it is invalidated when the source changes; until then there is a staleness window. Pick the smallest acceptable staleness for the data, prefer update-DB-then-delete-cache, and always keep a TTL backstop — because invalidation events are best-effort and concurrency makes a perfect cache→source sync impossible.
