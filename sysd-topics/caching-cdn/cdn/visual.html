CDN — The Request Flow

How a Content Delivery Network serves content from an edge server physically near the user — turning a ~150-250 ms cross-continent origin trip into a ~5-30 ms nearby edge hit, while shielding the origin from nearly all the traffic.

Real-world analogy — the publisher and the local bookstores

A popular book publisher sits in Mumbai (the origin server). Couriering every order straight from Mumbai to readers in Chennai, Delhi and Kolkata would take 3-4 days and overload the warehouse. Instead the publisher opens a local bookstore in each city (the edge servers / PoPs). The first Chennai reader to want a book triggers the store to fetch one copy from Mumbai and shelve it (cache MISS → origin fetch); every reader after that gets it from the local shelf in a day (cache HIT). Each book carries a "best before" date (TTL); once it expires the store checks with Mumbai whether a new edition exists. Bring the content to the user, and bother the origin only on cold misses.

No CDN — everyone hits the origin

every request → far origin

At 100k QPS the origin alone serves all 100k requests, each paying the full cross-continent ~150-250 ms RTT. A traffic spike or volumetric DDoS lands directly on the origin and can take it down.

With a CDN — edge absorbs the reads

95% HIT at edge → origin sees 5%

A 95% cache hit ratio means at 100k QPS the origin only sees ~5k. Most users get <10 ms edge HITs, and spikes / DDoS are absorbed by the edge layer before reaching the origin.

Things to know in the interview

  • TTL ≠ invalidation. TTL is passive expiry (Cache-Control: max-age=3600 — the object goes stale on its own). A purge/invalidation is an active push that evicts edges immediately; global purge propagates in seconds-to-minutes. To fix "user sees stale data," combine sensible TTLs + versioned URLs + targeted purge.
  • Revalidation saves bandwidth. On TTL expiry the edge sends If-None-Match (ETag) / If-Modified-Since; if unchanged the origin returns 304 Not Modified with no body and the edge just refreshes its TTL.
  • Hit ratio is the metric, not "we added a CDN." A unique query-string on every URL makes the cache key too narrow → near-zero hits → origin still overloaded. Cache key design (URL + selected headers, Vary, query handling) directly drives hit ratio.
  • Shield / mid-tier cache sits between edges and origin so all edges don't hammer the origin on a miss — and damps a cache-miss storm when many TTLs expire at once (mitigate with TTL jitter).
  • DNS routing ≠ exact geo. CDNs often route by the user's DNS resolver location, so a public resolver (8.8.8.8) can pick a suboptimal PoP. EDNS Client Subnet + Anycast reduce this gap.
  • Not just static. Edge compute (Cloudflare Workers, Lambda@Edge, Fastly Compute) and micro-caching (even a 1-second TTL) let dynamic and API responses be served from the edge. stale-while-revalidate serves the stale copy instantly while refreshing in the background.
  • Used by — Cloudflare, Akamai, AWS CloudFront, Fastly, Google Cloud CDN, Netflix Open Connect.

Key takeaway

A CDN is a geographically distributed cache layer: DNS/Anycast steers each user to the nearest edge PoP, which serves a cache HIT in <10 ms or, on a MISS, fetches once through a shield from the far origin and caches it for everyone else. TTL governs freshness, revalidation and purge keep it correct, and a high cache hit ratio is what offloads the origin — the real scalability win.
