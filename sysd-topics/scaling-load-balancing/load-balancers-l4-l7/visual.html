Load Balancers — L4 vs L7

How a load balancer spreads traffic across backends, and the core split: an L4 balancer routes on the 5-tuple (IP + port) without opening the packet, while an L7 balancer terminates TLS and reads the HTTP path/headers/cookies to route on content.

Real-world analogy — the airport gate

Thousands of passengers (requests) pour into an airport. An L4 load balancer is the security guard who just glances at the gate number on your boarding pass (IP + port) and waves you straight to that counter — fast, because they never open your bag. Once you're sent to a counter, your whole conversation stays there (connection sticky). An L7 load balancer is the smart concierge who actually talks to you — "where are you headed? business class? wheelchair?" — reading the content of your request (URL path, header, cookie) to pick the best specialised counter. Smarter, but talking to everyone costs a little more time. L4 reads the envelope; L7 opens the letter.

L4 — Transport layer

backend = hash(src/dst IP + port + proto)

Reads only the 5-tuple, forwards packets (NAT / Direct Server Return) without parsing the payload. TLS is pass-through — encrypted bytes flow end-to-end to the backend.

  • Latency added: ~µs (sub-millisecond)
  • Throughput: millions of packets/sec
  • Sticky at connection level — one TCP conn pinned to one backend
  • Protocol-agnostic: balances any TCP/UDP (DBs, brokers, gaming, VoIP)
  • e.g. AWS NLB, Google Maglev, LVS/IPVS

L7 — Application layer

backend = route(method, path, host, header, cookie)

Terminates TLS (decrypts), parses the HTTP request, then routes on content: /api/*→one pool, /static/*→another. New pooled connection to the backend.

  • Latency added: ~1–5 ms (TLS + parse cost)
  • Throughput: tens-to-hundreds of thousands req/sec
  • Sticky at request level — cookie-based session affinity
  • Enables retries, rewrites, canary, gRPC/HTTP-2 routing, WAF
  • e.g. AWS ALB, NGINX, HAProxy, Envoy

Things to know in the interview

  • "L7 is always better" is wrong. L7 adds features and latency + CPU. For DB connections or gRPC streaming, plain L4 is often the better choice. The right answer is "depends on the requirement."
  • L4 cannot do URL-based routing. It never sees the HTTP path (encrypted / un-parsed payload), so "send /api to a different backend on L4" is impossible — path routing is strictly an L7 thing.
  • Pass-through vs termination. L4 = TLS pass-through (forwards encrypted bytes). L7 = TLS termination (decrypts). To route on cookies/headers, L7 must terminate — you can't read encrypted content.
  • Deep vs shallow health checks. L4 checks are usually shallow (TCP connect succeeds?). L7 can do deep checks (GET /health → expect 200), catching backends that are listening but broken.
  • Stickiness vs even load. Sticky sessions (L4 connection-level or L7 cookie-based) can overload one server if a client is heavy — perfect even distribution and stickiness can't both be fully achieved. Name the tradeoff.
  • Layered in practice. Big systems put a global L4 in front (raw throughput, DDoS absorption) feeding regional L7 balancers (smart routing). Both layers together impresses interviewers.

Key takeaway

An L4 balancer routes on the 5-tuple (IP + port) without opening the packet — TLS passes through, latency is microseconds, and it balances any TCP/UDP. An L7 balancer terminates TLS and reads the HTTP path / headers / cookies to route on content — costing a few milliseconds but unlocking path routing, cookie affinity, retries, and WAF. Speed vs intelligence — and large systems use both, layered.
