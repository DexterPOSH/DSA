Reverse Proxy — The Request Journey

How a single front door (api.example.com) receives every client request, terminates TLS, routes by path, load-balances across a healthy backend pool, and serves cached responses — all while keeping backends hidden from clients.

Real-world analogy — the corporate reception desk

Picture the reception desk of a big office. A visitor can't just walk in and roam the floors — they meet the receptionist first. She receives everyone at one address, checks identity, and routes each visitor to the right person inside. The visitor never knows who's in, who's on leave, or which floor anyone sits on — they only know the reception address. The receptionist hides all the internal complexity: she load-balances (if one person is busy, send to another), does security, and controls a single entry point. A reverse proxy is that receptionist — clients know only api.example.com; the 50 backends behind it can come, go, or fail invisibly.

L4 load balancer (fast & dumb)

forwards TCP/UDP packets only

Works at the transport layer — it sees IPs and ports, not HTTP. So it can't read the URL path or Host header.

  • No path/host routing
  • No TLS termination, no caching, no header rewrite
  • Lower per-request overhead, very high throughput
  • e.g. AWS NLB

L7 reverse proxy (smart)

understands the HTTP request

Reads method, path, headers — so it can do everything the journey above shows. Load balancing is just one feature of it, not its identity.

  • Path/host routing (/api vs /static)
  • TLS termination + caching + gzip + rate limit
  • Slightly higher per-request cost
  • e.g. nginx, Envoy, HAProxy, AWS ALB

Things to know in the interview

  • Reverse ≠ forward. A reverse proxy sits in front of servers and hides backends from clients (client doesn't know which server). A forward proxy sits in front of clients and hides clients from servers (corporate egress / VPN). "Reverse" = the direction of who-it-fronts is flipped.
  • Single point of failure. Everything flows through the proxy — if it dies, the whole service dies. Always run it redundant: multiple instances behind a floating VIP / DNS failover / Anycast.
  • Client IP is lost. Backends see the proxy's IP, not the real client. The proxy adds X-Forwarded-For — and you should only trust it when it came from a trusted proxy (it's spoofable).
  • TLS termination tradeoff. Terminate at the proxy → it sees plaintext (routing, caching, WAF possible), but the proxy→backend hop must be re-encrypted (mTLS) on an untrusted network. Passthrough keeps end-to-end TLS but loses L7 features.
  • Caching is dangerous for auth'd content. A cache hit serves in microseconds vs a 10–100ms backend round-trip — but caching personalized/authenticated responses can leak one user's data to another. Handle cache keys / Vary carefully.
  • Sticky sessions. If backends keep session state locally, plain round-robin logs users out on their next request. Use session affinity — or better, externalize state (Redis) so backends stay stateless.
  • Variants. API Gateway (proxy + auth + rate limit + API composition), CDN edge (geo-distributed proxy + cache), sidecar proxy (Envoy per pod in a service mesh).

Key takeaway

A reverse proxy is the single front door that every client request passes through: it terminates TLS, routes by path/host, load-balances across a healthy backend pool, and caches — keeping backends hidden and centralizing all cross-cutting concerns. Load balancing is just one of its jobs, it must be run redundant so it isn't a single point of failure, and it adds the X-Forwarded-For header so backends still know the real client.
