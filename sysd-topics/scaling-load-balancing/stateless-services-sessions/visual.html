Stateless Services & Sessions

Why a stateless service lets any instance serve any request at any time — so a load balancer can fan out freely, autoscaling is trivial, and a dying pod loses nothing. The trick: where does the state live? Not in the instance's RAM — in the token or a shared Redis store.

Real-world analogy — the bank branch with two kinds of counters

Stateful counter (old way): you go to one specific teller who keeps your whole case file in their desk drawer. If that teller takes the day off (server crash) — or you must keep returning to the same teller (sticky session) — the system gets rigid. A new teller can't serve you without your file.

Stateless counter (modern way): every customer carries their own passbook + ID. Walk up to any free teller, show the passbook, done. The teller just reads it — they never have to remember you. Grow from 5 tellers to 50 and nothing breaks; whoever is free serves you. That is a stateless service: any instance, any request, anytime.

Sticky sessions (stateful + routing crutch)

LB pins client → always instance 2

Session lives in instance 2's RAM, so the LB must keep sending you back. Works… until instance 2 dies — then 100% of its sessions vanish and those users are logged out. Load also piles unevenly; a new pod gets no existing traffic.

True stateless (externalized state)

LB → any instance → state from token / Redis

No per-client RAM, so the LB routes freely (round-robin / least-conn). Kill any pod and the user never notices — their session is safe in Redis. Add pods to go 10K→50K QPS with no warm-up.

Opaque session ID + store

Cookie: sid=8f3a… → Redis GET sid
  • + Instant revocation — delete the key, session is dead
  • − One store round-trip per request (~1-2 ms, same-AZ)
  • − Store becomes a critical dependency (must replicate)

Self-contained JWT

Bearer eyJhbGci… {uid, roles, exp} signed
  • + No lookup — verify signature on CPU in microseconds
  • − Hard to revoke — valid until exp (use 5-15 min TTL + refresh token)
  • − 200 B – 1 KB travels on every request

Things to know in the interview

  • "Stateless" ≠ "no state." The state still exists — it's just externalized (Redis, DB, or the client's token). When asked "where does the state live?" the answer is "the store / the token," never "nowhere."
  • Sticky sessions are not stateless — they're stateful with a routing crutch. They reintroduce uneven load and lose all sessions on the instance that dies. Calling them a scaling solution is a red flag.
  • JWT revocation is the gotcha — you can't unilaterally kill a self-contained token. A blocklist works but re-adds a store + lookup (breaking the no-lookup win), so the real answer is short TTL + a revocable refresh token.
  • The store is a new single point of failure — you moved state, you didn't delete it. Replicate it (Redis Sentinel/Cluster), else "store down = everyone logged out."
  • Stateless ≠ idempotent — stateless = no per-client memory in the instance; idempotent = same request twice gives the same result. Different concepts; don't conflate them.
  • Where it shows up — 12-Factor App Factor VI, AWS ALB + Auto Scaling Groups, Kubernetes Deployments + HPA, and serverless (Lambda) all assume instances are interchangeable.

Key takeaway

Keep zero per-client state in an instance's RAM and push the real state into the token or a shared store (Redis). Then every instance is interchangeable: the load balancer fans out freely, autoscaling and rolling deploys are trivial, and a crashing pod loses nothing — at the cost of a sub-millisecond store lookup (or a bit of token weight) on each request.
