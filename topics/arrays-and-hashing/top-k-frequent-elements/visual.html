Why this is O(n), not O(n log n)

The heap approach pushes every distinct element through a size-k priority queue.

Min-heap of size k

O(n log k)

Each of the n distinct counts costs a log-k push/pop. Simple to write, but the log factor is real work.

Bucket sort

O(n)

A frequency is at most n, so there are only n + 1 buckets. Counting is O(n), filling buckets is O(n), and one linear scan collects the answer. No sorting, no log factor.

The key insight: we never need to sort frequencies — the bucket index itself is the frequency. Bounded keys (0…n) let us replace comparison sorting with direct array indexing.

Key takeaway

Because frequency is bounded by the array length, bucketing numbers by their count and reading the fullest buckets first finds the top k in O(n) — no heap, no sort.
